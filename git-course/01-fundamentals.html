Git Fundamentals

What Git is, why it works the way it does, and the three-area model every developer has internalized.

What is Git?

Git is a distributed version control system. Every developer has a complete copy of the entire project history — not just the latest files, but every snapshot that was ever saved. That's why you can work offline, create branches instantly, and roll back to any point in time.

Git was created by Linus Torvalds in 2005 to manage the Linux kernel. Its design priorities were speed, data integrity, and support for non-linear development (thousands of parallel branches).

ℹ️

Git ≠ GitHub. Git is the command-line tool that runs locally. GitHub is a cloud hosting service for Git repositories that adds collaboration features (PRs, Issues, Actions). Others exist: GitLab, Bitbucket, Azure DevOps.

Snapshots, not diffs

Most older VCS tools (SVN, CVS) store changes as deltas — "file X had line 5 changed from A to B." Git works differently: each commit is a snapshot of the entire project at a point in time. Unchanged files are referenced, not copied.

The Three-Area Model

This is the most important mental model in Git. Every file you work with lives in one of three places:

Understanding each area

📝

Working Directory

The actual files on your filesystem. When you edit a file in VS Code or vim, you're editing the working directory. Git knows the file changed but hasn't done anything with that change yet.

📋

Staging Area (Index)

A preparation zone. You explicitly choose which changes go into the next commit with git add. This lets you craft focused, logical commits even if your working directory is messy.

🗄️

Repository (.git)

The permanent history. Once a commit is here, it stays (unless you explicitly rewrite history). Contains all commits, branches, tags, and Git's object database — all stored in the hidden .git/ folder.

How Git stores data: Object types

Under the hood, Git uses a content-addressable key-value store. Everything is identified by a SHA-1 hash of its content. Four object types exist:

ObjectWhat it isExample SHA
blob Contents of a single file (no filename, no metadata) d670460...
tree A directory listing — maps filenames to blob or tree hashes 83baae6...
commit Points to a tree (root snapshot) + parent commit(s) + author + message 1a2b3c4...
tag A named pointer to a commit, usually a release (e.g. v2.1.0) abc1234...
🔍

TSE Tip: When a customer says "my code was working at commit abc1234," that SHA is pointing directly at a tree snapshot. You can checkout exactly that state with git checkout abc1234 — no hunting through diffs.

Git is distributed

Unlike centralized systems (SVN), every developer has a full clone of the repository, including the entire history. This means:

  • You can commit, branch, and view history offline
  • The server (GitHub) is just another clone — there's nothing magic about it
  • git push sends your commits to a remote; git pull brings remote commits to you
  • If GitHub goes down, every developer still has the full repo locally

First-time Git setup

Before committing anything, tell Git who you are:

# Set your identity globally (stored in ~/.gitconfig)
git config --global user.name "Your Name"
git config --global user.email "you@company.com"

# Set default branch name to "main" (modern default)
git config --global init.defaultBranch main

# Choose your editor (e.g. VS Code)
git config --global core.editor "code --wait"

# Verify your config
git config --list
⚠️

Your name and email are baked into every commit you make. They show up in git log, on GitHub, and in blame output. Use your real work email so commits are linked to your GitHub account.
