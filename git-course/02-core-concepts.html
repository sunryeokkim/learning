Core Concepts

Commits, branches, HEAD, merging, rebasing, and tags — the vocabulary you need to read any Git output.

Commits

A commit is an immutable snapshot saved to the repository. Once made, its SHA hash is fixed — you can't change a commit, only create new ones. Each commit stores:

  • A pointer to the root tree (the project snapshot)
  • One or more parent commit SHA(s)
  • Author name, email, and timestamp
  • Committer name, email, and timestamp (may differ from author after a rebase)
  • A commit message

Commits form a Directed Acyclic Graph (DAG): each commit points backward to its parents. This creates the immutable history chain.

Branches

A branch is simply a lightweight movable pointer to a commit. Creating a branch doesn't copy files — it just creates a 41-byte file in .git/refs/heads/ containing a SHA hash. That's why branch creation is instantaneous in Git.

HEAD

HEAD is a special pointer that tells Git "you are here." Usually it points to a branch name (like main), and the branch points to a commit. When you make a new commit, the branch pointer moves forward and HEAD moves with it.

Attached HEAD (normal)

HEAD → main → f1a2b3

You are on a branch. New commits advance the branch.

Detached HEAD

HEAD → 789abc (directly)

You checked out a specific commit. Any commits you make have no branch — they'll be orphaned if you switch away.

🔍

TSE scenario: When you git checkout <commit-SHA> to investigate a specific version, you enter detached HEAD state. That's fine for investigation — just don't commit anything there unless you create a branch first.

Merging

Merging brings two branches back together. Git handles this in two ways:

Fast-forward merge

If the target branch hasn't diverged, Git simply moves the pointer forward. No new commit is created.

Three-way merge (merge commit)

When both branches have diverged, Git finds their common ancestor, compares both tips to it, and produces a new merge commit with two parents.

Rebasing

Rebase is an alternative to merging that rewrites history to make it look linear. It replays your commits on top of another branch, creating new commit SHAs.

Merge

  • Preserves full history including branch topology
  • Creates a merge commit (extra node)
  • Safe on public/shared branches
  • git merge feature

Rebase

  • Creates a linear, cleaner-looking history
  • No merge commit
  • Rewrites SHAs — avoid on shared branches
  • git rebase main

Tags

Tags are named references to specific commits, typically used to mark release versions. Unlike branches, tags don't move when new commits are added.

TypeCommandWhat it storesUse case
Lightweight tag git tag v1.0 Just a pointer (like a branch) Quick local markers
Annotated tag git tag -a v1.0 -m "Release 1.0" Full object: tagger, date, message, GPG signature Official releases — always use this 
# List all tags
git tag

# See commit behind a tag
git show v1.0

# Check out the state of code at a release tag
git checkout v1.0

# Push tags to GitHub (not pushed automatically with git push)
git push origin --tags
🔍

TSE Tip: When a customer reports "it worked in v2.3.1 but broke in v2.4.0," use git checkout v2.3.1 to reproduce the working state and git log v2.3.1..v2.4.0 --oneline to see every commit between the two releases.

Remote tracking branches

When you clone a repo, Git creates remote-tracking references like origin/main. These are read-only snapshots of what the remote looked like last time you communicated with it.

# See all local + remote branches
git branch -a
  main
  feature/login
* remotes/origin/main
  remotes/origin/feature/login
  remotes/origin/bugfix/crash

# Update remote-tracking refs without merging
git fetch origin

# Fetch + merge in one step
git pull origin main
