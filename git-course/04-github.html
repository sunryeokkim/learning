Section 4

GitHub Features

The collaboration layer on top of Git — Pull Requests, Issues, Actions, Releases, and more.

Repositories

A GitHub repository is a hosted Git repo with extra metadata and collaboration features. Key anatomy:

GitHub elementWhat it is
Code tabThe file tree and commit history. You can browse files at any branch or tag.
IssuesBug reports, feature requests, tasks. Can be linked to PRs.
Pull RequestsProposed code changes. The main collaboration mechanism.
ActionsCI/CD workflows triggered by events (push, PR, schedule).
ReleasesTagged version snapshots with changelogs and attached artifacts.
Insights → NetworkVisual graph of all branches and forks.
Settings → BranchesBranch protection rules (required reviews, CI status checks).

Forks

A fork is a server-side copy of a repository under your own account. It's GitHub's way of enabling contributions to repos you don't have write access to.

Pull Requests (PRs)

A Pull Request is a request to merge one branch into another. It's the central unit of code collaboration on GitHub — where code review, discussion, and CI all converge.

PR Lifecycle

Reading a PR as a TSE

When investigating a customer issue, a PR tells you:

  • What changed: the "Files changed" tab shows the exact diff
  • Why it changed: the PR description and linked Issue
  • When it shipped: the merge date correlates with the release
  • Risks flagged: review comments and CI failures during the process
  • Rollback options: the "Revert" button creates an inverse PR

PR merge strategies

StrategyWhat happensHistory style
Merge commit Creates a merge commit with two parents Preserves branch history; shows topology
Squash and merge All PR commits combined into one commit on main Very clean linear history; loses individual commits
Rebase and merge Each PR commit replayed individually on main (new SHAs) Linear history; individual commits visible
🔍

TSE tip: If a team uses squash-merge, each PR becomes a single commit on main. The PR number is in the commit message (e.g. fix: token expiry (#4521)). Search GitHub for that issue number to find all context: discussion, diff, linked issue.

GitHub Actions (CI/CD)

GitHub Actions is an event-driven automation system. Workflows are YAML files in .github/workflows/ that run on GitHub-hosted runners (virtual machines).

Anatomy of a workflow

# .github/workflows/test.yml
name: Run Tests

on:                          # Trigger events
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest   # Runner OS

    steps:
      - uses: actions/checkout@v4        # Check out code
      - uses: actions/setup-python@v5    # Set up Python
        with:
          python-version: '3.12'
      - run: pip install -r requirements.txt
      - run: pytest                      # Run tests

Common Actions concepts for TSEs

TermMeaning
RunnerThe VM that executes the job (GitHub-hosted or self-hosted)
StepA single task: either a shell command (run:) or a pre-built action (uses:)
ActionA reusable unit of automation, published on GitHub Marketplace
ArtifactFiles produced by a job (test results, binaries) that can be downloaded
SecretEncrypted env vars (API keys, tokens) stored in repo/org settings
EnvironmentNamed deployment target (staging, production) with its own protection rules

Issues

GitHub Issues are the task/bug-tracking system. Key features:

  • Labels: categorize by type (bug, enhancement), priority (P1), component
  • Milestones: group issues into releases or sprints
  • Assignees: who is responsible
  • Linked PRs: typing Fixes #123 in a PR description auto-closes the issue when the PR merges

Releases

A GitHub Release is a snapshot tied to a Git tag, with:

  • A changelog (auto-generated from PR titles or hand-written)
  • Attached binary assets (compiled executables, Docker images, etc.)
  • A "pre-release" flag for beta/RC versions
# Create a release tag locally and push it
git tag -a v2.5.0 -m "Release v2.5.0 — OAuth support"
git push origin v2.5.0
# Then go to GitHub → Releases → "Draft new release from tag"
🔍

TSE tip: The GitHub Releases page is often the fastest way to answer "what changed between versions." The auto-generated changelog groups PRs by type if the team uses conventional commits or GitHub's release notes feature.

Branch Protection Rules

Configured under Settings → Branches, these rules prevent mistakes on important branches:

RuleEffect
Require pull request reviewsYou can't merge without N approved reviews
Require status checks to passCI must be green before merging
Require branches to be up to dateBranch must have latest main before merging
Restrict who can pushOnly specified roles/teams can push directly
Require signed commitsAll commits must have a GPG/SSH signature

GitHub Search

GitHub's search is powerful — TSEs should know the key qualifiers:

# Search in a specific repo
repo:datadog/datadog-agent "connection refused"

# Find PRs that touched a specific file
repo:org/repo path:src/auth.py is:pr is:merged

# Issues labeled bug, closed in last 30 days
repo:org/repo is:issue is:closed label:bug closed:>2024-01-01

# Find commits containing a keyword
repo:org/repo "fix memory leak" is:commit
