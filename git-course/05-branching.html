Section 5

Branching Strategies

The three most common workflows teams use — knowing which one a customer uses tells you how to map their release pipeline.

🔍

Why TSEs need this: When a customer says "the bug is in the release branch," you need to know what that means in their branching model. Understanding the strategy also tells you how hotfixes flow and how long a bug might be present before it reaches production.

GitHub Flow

The simplest strategy. One long-lived branch (main), and every change goes through a short-lived feature branch + PR. Teams that deploy continuously (multiple times per day) use this.

GitHub Flow rules

  • main is always deployable
  • Any work happens in a descriptively-named branch
  • PR = request for code review
  • After review and CI pass, merge to main and deploy
💡

Best for: Web apps with continuous deployment, small-to-medium teams, products that don't need to maintain multiple versions simultaneously.

GitFlow

A more structured model with multiple permanent branches and strict rules about where different types of changes go. Created by Vincent Driessen in 2010. Common in projects that ship discrete versioned releases (mobile apps, packaged software, SDKs).

GitFlow branch roles

BranchPurposeLong-lived?
main (or master)Production-ready code only. Tagged at each release.Yes
developIntegration branch. Contains completed features waiting for the next release.Yes
feature/*Individual features. Branch from develop, merge back to develop.No
release/*Release preparation: only bug fixes, docs, version bumps. Branch from develop, merge to both main and develop.No
hotfix/*Urgent production fixes. Branch from main, merge to both main and develop.No
💡

Best for: Software with scheduled release cycles, multiple maintained versions (v1.x and v2.x simultaneously), mobile apps, or any project where "we can't push to production every hour."

Trunk-Based Development (TBD)

All developers commit frequently (multiple times per day) directly to main (the "trunk"), or use very short-lived feature branches (max 1-2 days). Changes are kept small and incomplete features are hidden behind feature flags.

Feature flags in TBD

Incomplete features land in main but are wrapped in a flag check:

# The code is deployed but inactive until the flag is enabled
if feature_flags.is_enabled("new_search_ui", user):
    return render_new_search()
else:
    return render_legacy_search()
💡

Best for: Large engineering teams (Google, Facebook, Netflix all use TBD), continuous deployment, teams that can invest in feature flagging infrastructure and comprehensive test coverage.

Which model is this customer using?

Quick diagnostic questions to identify a customer's branching strategy:

SignalLikely model
"We have a develop branch and release/ branches" GitFlow
"We deploy whenever a PR merges to main" GitHub Flow
"We push to main multiple times a day and use feature flags" Trunk-Based
"We have long-lived release/1.x branches with backport PRs" GitFlow or variant
"Our main branch is always green and always deployed" GitHub Flow or Trunk

Hotfixes and backports

When a critical bug is found in production, it needs to be fixed fast and deployed — bypassing the normal feature pipeline.

In GitFlow:

  1. 1
    Branch hotfix/issue-desc from main (not develop)
  2. 2
    Make the fix, bump the patch version
  3. 3
    Merge into main → tag a new patch release (v2.0.1)
  4. 4
    Also merge into develop so the fix isn't lost in the next release

Backporting (multiple maintained versions)

If a team maintains both v2.x and v3.x, a fix merged to main (v3.x) must be backported to the v2.x release branch:

# Cherry-pick the specific fix commit to the old release branch
git checkout release/2.x
git cherry-pick f1a2b3   # the fix commit SHA from main
git push origin release/2.x
# Then open a PR for the backport on GitHub
