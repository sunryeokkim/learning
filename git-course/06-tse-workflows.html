Section 6

TSE Workflows

Hands-on playbooks for the most common Git tasks you'll face investigating support tickets.

Workflow 1: Reproducing a bug at a specific version

Customer says: "It was working on v2.3.1 but broke in v2.4.0."

  1. 1
    Clone the repo (shallow to save time) 
    git clone --filter=blob:none https://github.com/org/repo.git
cd repo
  2. 2
    List available release tags to confirm versions exist 
    git tag --sort=-version:refname | head -20
v2.4.1
v2.4.0
v2.3.2
v2.3.1   ← working version
v2.3.0
  3. 3
    Check out the broken version 
    git checkout v2.4.0
# You are in 'detached HEAD' state — that's fine for investigation
  4. 4
    Reproduce the bug, then switch to the working version 
    git checkout v2.3.1
# Verify the bug is absent here
  5. 5
    See what changed between versions 
    # All commit messages between v2.3.1 and v2.4.0
git log v2.3.1..v2.4.0 --oneline

# Only commits that touched the suspected file
git log v2.3.1..v2.4.0 --oneline -- src/auth/token.py

# Full diff between the two releases
git diff v2.3.1 v2.4.0 -- src/auth/token.py

Workflow 2: Finding the commit that introduced a bug (git bisect)

git bisect uses binary search to find the exact commit that introduced a regression. You only need to know a "good" commit and a "bad" one — Git does the rest.

# Start bisect session
git bisect start

# Mark current HEAD as bad (the bug exists here)
git bisect bad

# Mark a known-good commit or tag
git bisect good v2.3.1

Bisecting: 23 revisions left to test (roughly 5 steps)
[abc1234] feat: add caching layer

# Git checks out a midpoint. Run your test / reproduce the bug.
# If bug is present:
git bisect bad

# If bug is absent:
git bisect good

# Repeat until Git announces the culprit:
f1e2d3 is the first bad commit
Author: Bob <bob@example.com>
Date: Mon Mar 4 14:22:31 2024
    feat: refactor token validation

# End the session (returns to HEAD)
git bisect reset

# Automate with a test script (exits 0 = good, non-zero = bad)
git bisect run pytest tests/test_auth.py

Workflow 3: Tracing a code change (git blame + log)

Customer asks: "Who changed the timeout value and why?"

# Find which commit last touched a specific line
git blame src/config.py
^a1b2c3 (Alice 2024-01-10 09:00) TIMEOUT = 30
 f7e8d9 (Bob   2024-03-15 14:30) MAX_RETRIES = 5   ← changed by Bob on Mar 15

# Inspect that specific commit
git show f7e8d9
commit f7e8d9...
Author: Bob <bob@example.com>
Date:   Fri Mar 15 14:30:00 2024

    fix: reduce retries to avoid cascade failures

    Prior to this change MAX_RETRIES=10 caused thundering-herd
    under high load. Reducing to 5 per discussion with SRE team.

diff --git a/src/config.py b/src/config.py
-MAX_RETRIES = 10
+MAX_RETRIES = 5

# Find the PR that merged this commit (GitHub: search by SHA)
# Or use git log to find nearby merge commits
git log --merges --ancestry-path f7e8d9..HEAD --oneline | head -3

Workflow 4: Investigating a recent deployment

An alert fires 10 minutes after a deploy. Find what changed.

# Find commits in the last 24 hours on main
git log origin/main --since="24 hours ago" --oneline

# Find commits between two deploy timestamps
git log --after="2024-03-15 14:00" --before="2024-03-15 15:30" --oneline

# Find which files changed in the last deploy
git diff --name-only HEAD~5..HEAD

# Quick summary of what changed (number of changes per file)
git diff --stat HEAD~5..HEAD
 src/api/handler.go    | 42 ++++++++++--
 src/config/timeout.go |  8 ++--
 go.sum                |  2 +-
 3 files changed, 47 insertions(+), 5 deletions(-)

Workflow 5: Checking if a fix is included in a release

Customer asks: "Is fix #4521 in version 2.6.0?"

# Method 1: check if the commit SHA is an ancestor of the tag
git merge-base --is-ancestor abc1234 v2.6.0 && echo "YES, included" || echo "NOT included"

# Method 2: search log between tags
git log v2.5.0..v2.6.0 --oneline --grep="fix #4521"

# Method 3: check which tags contain the commit
git tag --contains abc1234
v2.6.0
v2.6.1
v2.7.0

# Method 4: find the earliest tag containing the commit
git tag --contains abc1234 --sort=version:refname | head -1
v2.6.0    ← first release that includes this fix

Workflow 6: Understanding the repo structure quickly

You just cloned an unfamiliar repo. Orient yourself fast.

# See recent activity — who's been working on what
git log --oneline --all --since="30 days ago"

# List all branches (local + remote)
git branch -a

# Visualize branch topology
git log --oneline --graph --all --decorate

# List all tags (releases)
git tag --sort=-version:refname | head -10

# See the most frequently changed files (active hotspots)
git log --name-only --format="" | sort | uniq -c | sort -rn | head -20

# List contributors
git shortlog -sn --all
   452  Alice Smith
   289  Bob Jones
    98  Carol Williams

Workflow 7: Working with a customer's fork or private repo

When a customer shares a private GitHub repo or their own fork:

# Clone their repo directly
git clone https://github.com/customer/their-fork.git

# Add their remote alongside the original upstream
git remote add customer https://github.com/customer/their-fork.git
git fetch customer

# See differences between upstream and customer's version
git log origin/main..customer/main --oneline
# ^ commits in customer's fork NOT in upstream

# Compare a specific file
git diff origin/main customer/main -- path/to/config.yaml

Workflow 8: Cherry-picking a fix to test

A fix was merged to main but the customer is on an older release. You want to test if the fix works for them.

# Check out the customer's version
git checkout -b test/backport-fix v2.3.1

# Apply just the fix commit on top
git cherry-pick f1a2b3

# If conflicts arise, resolve them and continue
git cherry-pick --continue

# Verify the fix works, then clean up
git checkout main
git branch -D test/backport-fix
🔍

Communication tip: When you find a fix commit via git bisect or git blame, always check the linked GitHub PR. The PR description usually explains the full context, links to the bug, and shows the discussion — all of which helps write a much better ticket response.
