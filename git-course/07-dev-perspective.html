Section 7

Dev Perspective

How developers actually use Git day-to-day — so you can speak their language and understand why history looks the way it does.

ℹ️

This section bridges the gap between "I know the commands" and "I understand how a developer's mental model works." Knowing this helps you ask better questions when escalating and interpret commit history correctly.

A developer's daily Git routine

Here's what a typical developer does every morning:

# 1. Sync with the remote before starting work
git switch main
git pull --rebase origin main

# 2. Create a feature branch
git switch -c feature/JIRA-1234-add-rate-limiting

# 3. Work... commit frequently in small chunks
git add -p        # stage specific hunks, not whole files
git commit -m "feat: add rate limit middleware"

# 4. Push branch and open PR
git push -u origin feature/JIRA-1234-add-rate-limiting

# 5. Update the PR after code review feedback
git add -p
git commit -m "review: rename middleware per feedback"
git push

Interactive Rebase — history cleanup

Before merging a PR, developers often clean up their commits to tell a coherent story. git rebase -i (interactive rebase) lets you reorder, squash, rename, or drop commits.

# Rebase the last 4 commits interactively
git rebase -i HEAD~4

# Git opens your editor with a list of commits:
pick a1b2c3 feat: add rate limit middleware
pick d4e5f6 wip
pick 789abc fix typo in comment
pick f1e2d3 review: rename middleware per feedback

# You can change "pick" to:
#   squash (s) — combine with previous commit
#   fixup  (f) — combine, discard this commit's message
#   reword (r) — keep commit, edit message
#   drop   (d) — delete this commit
#   edit   (e) — pause and amend the commit

# After editing:
pick a1b2c3 feat: add rate limit middleware
fixup d4e5f6 wip
fixup 789abc fix typo in comment
fixup f1e2d3 review: rename middleware per feedback
# Result: 4 commits become 1 clean commit
⚠️

Interactive rebase rewrites history (creates new SHAs). If you've already pushed the branch, you'll need to git push --force-with-lease to update the remote. Never rebase main or other shared branches.

Code Review rituals

When a developer opens a PR, the review process follows a predictable pattern:

Review ActionGitHub UIWhat it means
Comment 💬 on a line Question or suggestion — no blocking intent
Request changes 🔴 "Changes requested" The PR should not merge until specific issues are addressed
Approve ✅ "Approved" Reviewer is satisfied; PR can merge (if branch protection requires N approvals)
Dismiss review admin action Clear a stale review — usually when a reviewer is unavailable

What developers check in code review

  • Correctness: Does this solve the problem? Edge cases covered?
  • Tests: Are there adequate tests? Do they actually fail before the fix?
  • Performance: Any N+1 queries, unbounded loops, memory leaks?
  • Security: Input validation, auth checks, secret exposure?
  • Readability: Variable names, code structure, necessary comments?
  • API contract: Any breaking changes to public interfaces?

Semantic Versioning (SemVer)

Most software uses MAJOR.MINOR.PATCH versioning (e.g. v2.4.1):

PartIncrement when…Example
MAJOR Incompatible API/behavior changes (breaking change) v2.0.0 → v3.0.0
MINOR New feature, backward-compatible v2.4.0 → v2.5.0
PATCH Bug fixes, no new features v2.4.0 → v2.4.1
🔍

TSE tip: If a customer upgraded from v2.3.x to v3.0.0 and something broke, it's likely a breaking change in the major version bump. Check the release notes / CHANGELOG and migration guide for that version.

Conventional Commits

Many teams enforce a structured commit message format. This powers automated changelogs, semantic-release tools, and makes git log much easier to read.

# Format: type(scope): description
# Body and footer are optional

feat(auth): add OAuth2 PKCE flow

Previously only supported implicit flow. PKCE is required
for mobile apps and SPAs per RFC 7636.

Closes #4521
BREAKING CHANGE: removed deprecated /oauth/implicit endpoint

Development team signals TSEs should know

Reading a PR to understand risk

When a customer reports a regression, check the PR that introduced it for these risk signals:

  • Large diff (+500 lines): higher chance of unintended side effects
  • No tests added: the change wasn't validated automatically
  • Many reviewers requested changes: there was controversy
  • "Do not merge" label or "WIP" in title: was merged prematurely
  • CI was red at merge time: branch protection might have been bypassed
  • Short time between open and merge: rushed review

CHANGELOG / Release Notes

Most projects maintain a CHANGELOG.md at the repo root. It's the fastest place to find what changed in each version — faster than reading every commit.

cat CHANGELOG.md
## [2.4.0] - 2024-03-15

### Added
- OAuth2 PKCE flow for mobile clients (#4521)
- Rate limiting middleware with Redis backend (#4489)

### Fixed
- Token expiry not propagated to downstream services (#4501)

### Breaking Changes
- Removed deprecated /oauth/implicit endpoint
