Section 8

Troubleshooting Common Issues

Fixing detached HEAD, merge conflicts, wrong commits, and other situations that cause panic.

💡

Golden rule before anything scary: Create a backup branch first. git branch backup-$(date +%s) costs nothing and saves you if you make a mistake.

Problem: Detached HEAD state

Symptom:

HEAD detached at f1a2b3
nothing to commit, working tree clean

Cause: You checked out a commit, tag, or remote branch directly (not a local branch). Any commits you make here have no branch and can be lost.

Fix:

# Option 1: Just go back to your branch (discard any commits you made)
git switch main

# Option 2: Save your detached commits by creating a branch
git switch -c my-experiment-branch
# Now your commits are safe on this new branch

Problem: Merge conflicts

Symptom:

CONFLICT (content): Merge conflict in src/auth.py
Automatic merge failed; fix conflicts and then commit the result.

Git couldn't automatically combine both versions of a file. You must manually resolve the conflict.

The conflict markers:

<<<<<<< HEAD
    timeout = 30  # our version
=======
    timeout = 60  # incoming version
>>>>>>> feature/increase-timeout

Fix:

  1. 1
    Edit the file — delete the conflict markers and keep the correct code
  2. 2
    Stage the resolved file and complete the merge 
    git add src/auth.py
git merge --continue
# Or: git commit (if --continue isn't available)
  3. 3
    To abort and go back to before the merge: 
    git merge --abort
💡

Use a visual merge tool: git mergetool opens your configured editor (VS Code's merge editor is excellent). Set it with git config --global merge.tool vscode.

Problem: Committed to the wrong branch

You made a commit on main but it should have been on your feature branch.

# Step 1: Note the SHA of the commit you need to move
git log --oneline -3
abc1234 feat: my new feature   ← this needs to move
def5678 previous commit
...

# Step 2: Create/switch to the correct branch (it includes the commit)
git switch -c feature/my-feature

# Step 3: Go back to main and remove the commit
git switch main
git reset --hard HEAD~1
# main now points to def5678; feature/my-feature still has abc1234
⚠️

Only do git reset --hard if you haven't pushed the commit to a shared remote yet. If you already pushed, use git revert instead to create an undo commit.

Problem: Need to undo a commit already pushed

Never git reset --hard + force push on a shared branch — you rewrite history other people have. Use git revert instead.

# Find the commit to undo
git log --oneline
f1a2b3 feat: bad change we need to undo
d4e5f6 previous good commit

# Create an inverse commit (safe for shared branches)
git revert f1a2b3
# Git opens editor for the revert commit message
# A new commit "Revert 'feat: bad change...'" is created

git push

Problem: I think I lost commits (reflog rescue)

You ran git reset --hard or deleted a branch and now commits seem gone. Don't panic — Git keeps a log of everywhere HEAD has pointed for 90 days.

# See everywhere HEAD has been
git reflog
f1a2b3 HEAD@{0}: reset: moving to HEAD~3
abc1234 HEAD@{1}: commit: feat: my work
def5678 HEAD@{2}: commit: more work
789abc HEAD@{3}: checkout: moving from main to feature/xyz

# Recover by checking out the lost commit SHA
git checkout abc1234

# Save it on a new branch
git switch -c rescued-work

Problem: "Your branch has diverged"

Your branch and 'origin/main' have diverged,
and have 2 and 3 different commits each, respectively.

This happens when both you and the remote have new commits since the last sync. You need to reconcile them.

# Option 1: Merge remote changes into yours (creates merge commit)
git pull origin main

# Option 2: Rebase your commits on top of remote (linear history)
git pull --rebase origin main

# If you caused this by force-pushing, fetch and reset to remote
git fetch origin
git reset --hard origin/main
# ⚠ This discards your local commits!

Problem: Wrong commit message

# Fix the most recent commit message (before pushing)
git commit --amend -m "fix: correct description"

# Fix an older commit (interactive rebase)
git rebase -i HEAD~3
# Change "pick" to "reword" on the commit you want to fix

Problem: Accidentally staged or committed a secrets file

🚨

If you pushed a secret (API key, password) to a public repo, assume it is compromised. Rotate the credential immediately — even after you delete it from history, it may have been scraped by bots within minutes.

# Remove from staging only (keep the file)
git restore --staged secrets.env

# Remove from last commit (before push)
git rm --cached secrets.env
git commit --amend

# Add to .gitignore so it never happens again
echo "secrets.env" >> .gitignore
git add .gitignore
git commit -m "chore: ignore secrets file"

# If already pushed, use git-filter-repo to purge history
# (requires pip install git-filter-repo)
git filter-repo --invert-paths --path secrets.env

Common GitHub authentication errors

ErrorCauseFix
Permission denied (publickey) SSH key not added to GitHub or SSH agent Add key: ssh-add ~/.ssh/id_ed25519 and add public key in GitHub Settings → SSH keys
remote: Support for password authentication was removed GitHub removed password auth in Aug 2021 Use a Personal Access Token (PAT) as the password, or switch to SSH
repository not found Wrong URL, private repo, or no access Verify the URL, check repo visibility, confirm you have at least Read access
Updates were rejected because the remote contains work... Remote has commits your local branch doesn't have Pull first: git pull --rebase origin main

Useful mental models for unsticking yourself

🔍

When confused: git status first

Always shows what Git thinks is happening and often suggests the exact command to fix it.

🕐

Commits are permanent

Even after "deleting," commits live in reflog for 90 days. You can almost always recover.

🌿

Branches are cheap

When in doubt, branch. Creating a backup branch before any risky operation costs nothing.

🔄

revert > reset for shared work

If the commit was pushed, revert creates a new undo commit. Never rewrite shared history.
