Module 01

Python Fundamentals

The building blocks of Python — types, numbers, strings, and the Python mental model that makes the language behave differently from C++/Java.

The Python Mental Model

Before writing a single line, understand this: everything in Python is an object. Variables are not boxes that hold values — they are labels that point to objects in memory. This single idea explains most Python "surprises."

Memory model
x = [1, 2, 3] → x ──────► [ list object @ 0x7f3a ] y = x → y ──────► [ same list object! ] y.append(4) → x is also [1, 2, 3, 4] ← both labels point to same object z = x[:] → z ──────► [ NEW list @ 0x7f9b ] ← slice creates a copy z.append(5) → x is still [1, 2, 3, 4] ← independent
⚠️
The most common bug in coding problems

When you do result.append(path) inside backtracking, you append a reference. If you later mutate path, result changes too. Always append a copy: result.append(path[:]) or result.append(list(path)).

Numeric Types

Integers — Arbitrary Precision

Python integers have no overflow. They grow as large as your RAM allows. This is a superpower for combinatorics and cryptography problems — you never need to worry about 64-bit overflow like in C++.

# Python int is arbitrary precision — no overflow ever
x = 2 ** 1000          # works fine (a 302-digit number)
factorial = 1
for i in range(1, 100):
    # WHY: multiply into accumulator rather than call math.factorial —
    # loop invariant: after iteration i, factorial == i!
    factorial *= i      # 100! has 158 digits — no problem

# Integer division — TWO operators exist, know the difference
10 / 3      # → 3.3333...  (always returns float, even 10/2 = 5.0)
10 // 3     # → 3          (// floors toward -∞, not toward 0)
-7 // 2     # → -4         # WHY -4 not -3: floor(-3.5) = -4 (toward -∞)
-7 % 2      # → 1          # WHY 1 not -1: Python guarantees remainder >= 0 when divisor > 0

# Useful identity: a == (a // b) * b + (a % b)  — holds for all integers including negatives
divmod(-7, 2)   # → (-4, 1) — returns both at once; cheaper than computing // and % separately
Python floor (//) vs C++/Java truncation for negative numbers
Python -7 // 2 C++ / Java -7 / 2 ───────────────────────────────────────────────────── Number line: ←────────────────────────────────────────────→ -4 -3.5 -3 -2 -1 0 │ ↑ │ │ exact result │ │ (-7 ÷ 2) │ │ └─── truncate toward 0 → -3 └───────────────────── floor toward -∞ → -4 Python: -7 // 2 = -4 -7 % 2 = 1 verify: (-4)×2 + 1 = -7 ✓ C++: -7 / 2 = -3 -7 % 2 = -1 verify: (-3)×2 + (-1) = -7 ✓
Python floors toward −∞; C++/Java truncate toward 0. The difference only appears for negative numbers.
# Bit operations — useful for DP bitmask problems
5 & 3       # → 1    AND:  0101 & 0011 = 0001
5 | 3       # → 7    OR:   0101 | 0011 = 0111
5 ^ 3       # → 6    XOR:  0101 ^ 0011 = 0110  (bit differs → 1)
~5          # → -6   # WHY -(n+1): two's-complement flips all bits, so ~n = -(n+1)
1 << 3      # → 8    # WHY: left-shift by k multiplies by 2^k; faster than ** for powers of 2
16 >> 2     # → 4    # WHY: right-shift by k is integer divide by 2^k (floors toward -∞ for negatives)
n & (n-1)   # → clears lowest set bit  # WHY: subtracting 1 flips trailing 0s and the lowest 1; AND masks them off
            # → if result == 0, n is a power of 2 (exactly one bit was set)
n & (-n)    # → isolates lowest set bit  # WHY: -n in two's complement flips all bits then +1, leaving only the lowest set bit

Float — When to Use and When to Avoid

Floats use IEEE-754 double precision (64-bit). They have ~15 significant digits of precision, but suffer from rounding errors. In competitive programming, prefer integer arithmetic whenever possible.

# Floating point trap
0.1 + 0.2           # → 0.30000000000000004  (not 0.3!)
# WHY: 0.1 and 0.2 have no exact binary representation; IEEE-754 stores nearest approximation
0.1 + 0.2 == 0.3    # → False  ← never compare floats with ==

# Safe float comparison
# WHY 1e-9: generous enough to absorb accumulated rounding, tight enough for most problems
abs(0.1 + 0.2 - 0.3) < 1e-9  # → True  ✓

# Useful float constants
float('inf')    # positive infinity — beats any real number; safe initial value for a running minimum
float('-inf')   # negative infinity — less than any real number; safe initial value for a running maximum
# WHY use float('inf') instead of a large integer like 10**18?
# float('inf') participates in comparisons correctly without knowing the input range
math.inf        # same as float('inf'), requires import math

# Integer square root (avoids float entirely)
import math
math.isqrt(17)  # → 4   (floor of √17, exact integer)
math.isqrt(16)  # → 4
# WHY prefer isqrt over int(n**0.5): for large n (e.g. n=10**18), float loses precision
# and int(n**0.5) may return n-1 or n+1 instead of the correct floor
int(16**0.5)    # → 4   but can fail for large n due to float imprecision!
💡
Use math.isqrt() not int(n**0.5)

For large values of n, int(n**0.5) can return the wrong answer because the float computation loses precision. math.isqrt() computes the exact integer square root.

Strings

Strings in Python are immutable sequences of Unicode characters. You cannot change a character in-place — every "modification" creates a new string object. This has big implications for performance.

s = "algorithm"
#    0123456789   (positive indices)
#   -9-8-7-6-5    (negative indices from end)

# Basic access
s[0]        # 'a'   — first character
s[-1]       # 'm'   — last character (same as s[len(s)-1])
# WHY negative indices: -k is equivalent to len(s)-k; avoids computing length explicitly
s[-2]       # 'h'

# Slicing  [start:stop]  — stop is EXCLUSIVE
# WHY exclusive stop: s[a:b] has length b-a, and ranges tile cleanly: s[:k] + s[k:] == s
s[0:4]      # 'algo'   — characters 0,1,2,3
s[4:]       # 'rithm'  — from index 4 to end (omitting stop = go to end)
s[:4]       # 'algo'   — from start to index 3 (omitting start = begin at 0)
s[:]        # 'algorithm' — shallow copy; on lists, use this to avoid aliasing bugs

# Step slicing  [start:stop:step]
s[::2]      # 'agrtm'  — every other character (step skips indices)
s[::-1]     # 'mhtirogla' — WHY: negative step reverses traversal direction
s[1:8:3]    # 'lr'   — indices 1, 4, 7

# Slicing never raises IndexError — out-of-range is clamped
# WHY: Python clamps start/stop to [0, len(s)], so no need for bounds checks around slices
s[0:100]    # 'algorithm' — same as s[:]

String Methods — The Complete Reference

s = "  Hello, World!  "

# ── Whitespace ──────────────────────────────────────────
s.strip()           # "Hello, World!"    — both ends; strips spaces, \t, \n
s.lstrip()          # "Hello, World!  "  — left only
s.rstrip()          # "  Hello, World!"  — right only

# ── Case ────────────────────────────────────────────────
s.lower()           # "  hello, world!  "
s.upper()           # "  HELLO, WORLD!  "
# WHY capitalize acts on the full string including leading spaces:
# it lowercases everything then uppercases the very first character (which is a space here)
s.capitalize()      # "  hello, world!  " → first char upper (after strip in practice)
s.title()           # "  Hello, World!  "
s.swapcase()        # swap upper↔lower

# ── Searching ───────────────────────────────────────────
s.find("World")     # 9 (first index) — returns -1 if not found (never raises)
s.rfind("l")        # 11 (rightmost)  — returns -1 if not found
# WHY choose index() over find(): use when absence is a bug; saves an explicit -1 check
s.index("World")    # 9 — like find() but raises ValueError if not found
s.count("l")        # 3 — non-overlapping occurrences
"World" in s        # True — prefer 'in' for boolean; cleaner and equally O(n)

# ── Checking ────────────────────────────────────────────
"123".isdigit()     # True — all chars are digits
"abc".isalpha()     # True — all chars are letters
"abc123".isalnum()  # True — letters or digits
"   ".isspace()     # True
"Hello".istitle()   # True
"HELLO".isupper()   # True
"hello".islower()   # True

# ── Splitting & Joining ─────────────────────────────────
"a,b,c".split(",")          # ['a', 'b', 'c']
# WHY split() with no arg differs from split(" "): no-arg collapses consecutive whitespace
"hello world".split()       # ['hello', 'world'] — splits on any whitespace, strips leading/trailing
"hello world".split(" ", 1) # ['hello', 'world'] — max 1 split (second part is unsplit remainder)
",".join(["a", "b", "c"])   # "a,b,c"  — WHY join on str not list: strings own the separator
"".join(["h","i"])           # "hi"     — use "" as separator for concatenation

# ── Replacement ─────────────────────────────────────────
"aabbcc".replace("b", "X")      # "aaXXcc" — all occurrences; returns new string (strings are immutable)
"aabbcc".replace("b", "X", 1)   # "aaXbcc" — count arg limits replacements; useful to replace only first

# ── Alignment ───────────────────────────────────────────
"hi".ljust(10)          # "hi        "
"hi".rjust(10)          # "        hi"
"hi".center(10)         # "    hi    "
# WHY zfill not rjust: zfill handles sign correctly ("−42".zfill(5) → "−0042")
"42".zfill(5)           # "00042"  — zero-pad to width

String Building — The Performance Trap

Strings are immutable. Doing result += char inside a loop creates a new string every iteration, making it O(n²) total. Always collect pieces in a list, then join at the end.

# ❌ SLOW — O(n²) because each += creates a new string
result = ""
for char in some_list:
    # WHY O(n²): strings are immutable, so each += allocates and copies the whole string so far
    result += char          # creates n new string objects; total bytes copied = 1+2+…+n = O(n²)

# ✓ FAST — O(n) — collect in list, join once
parts = []
for char in some_list:
    parts.append(char)      # list append is amortized O(1)
result = "".join(parts)     # join pre-allocates exact final size, copies once — O(n) total

# ✓ Even simpler with list comprehension
# WHY generator expression works here: join consumes it in one pass, no intermediate list needed
result = "".join(char for char in some_list if some_condition)

# f-strings — preferred for formatting (Python 3.6+)
name = "Alice"
score = 99.7
f"Player {name} scored {score:.1f}"   # "Player Alice scored 99.7"  (.1f = 1 decimal place)
f"{1000000:,}"                         # "1,000,000"  (comma as thousands separator)
f"{42:08b}"                            # "00101010"   (08 = pad to 8 wide with zeros; b = binary)
f"{255:02x}"                           # "ff"         (x = lowercase hex; 02 = at least 2 digits)

Characters and ASCII — Essential for Coding Problems

# ord() converts char to its Unicode code point (integer)
# chr() converts integer back to character

ord('a')    # 97   — lowercase letters occupy 97–122
ord('A')    # 65   — uppercase letters occupy 65–90 (gap of 32 from lowercase)
ord('0')    # 48   — digit characters occupy 48–57
chr(97)     # 'a'
chr(65)     # 'A'

# Map letter to 0-25 index (very common pattern)
# WHY subtract ord('a'): normalizes 'a'→0, 'b'→1, …, 'z'→25, enabling fixed-size array indexing
ord('e') - ord('a')   # 4  — 'e' is the 5th letter (0-indexed)
ord('E') - ord('A')   # 4  — same offset works for uppercase because A–Z are also contiguous

# Count letter frequencies using array instead of dict
# WHY array not dict: O(1) access, O(26) space regardless of input, cache-friendly
freq = [0] * 26
for c in "hello":
    freq[ord(c) - ord('a')] += 1
# freq[7] == 1 ('h'), freq[4] == 1 ('e'), freq[11] == 2 ('l'), freq[14] == 1 ('o')

# Check if anagram (same letter counts)
def is_anagram(s, t):
    # WHY early length check: two strings of different lengths can't be anagrams;
    # avoids the all() scan when the answer is already obvious
    if len(s) != len(t):
        return False
    count = [0] * 26
    for c in s: count[ord(c) - ord('a')] += 1   # increment for every char in s
    for c in t: count[ord(c) - ord('a')] -= 1   # decrement for every char in t
    # Loop invariant: after both loops, count[i] == freq_s[i] - freq_t[i]
    # WHY all(x == 0): if any bucket is nonzero, one string has a surplus letter the other doesn't
    return all(x == 0 for x in count)

# Common patterns using truthiness
nums = [1, 2, 3]

# WHY `if nums` instead of `if len(nums) > 0`: more Pythonic and avoids the len() call;
# works for any container (list, dict, set, deque) without knowing its type
if nums:
    process(nums)

node = TreeNode(0)

    ...           # Use `if node is not None:` if you need to distinguish val=0

# Short-circuit evaluation — Python evaluates left-to-right and stops early
a = None
# WHY `or` for defaults: `a or default` returns first truthy value, or last value if all falsy

# WHY use `and` to guard a call: if the guard is falsy, the expensive call is skipped entirely
d = [] and process()  # process() is NEVER called ([] is falsy → short-circuit)

Equality vs Identity

Python has two comparison operators. Mixing them up is a classic bug.

# == checks VALUE equality  (calls __eq__)
# is checks IDENTITY (same object in memory — same id())

a = [1, 2, 3]
b = [1, 2, 3]
c = a          # WHY: assignment copies the reference, not the list; c and a point to the same object

a == b      # True   — same values
a is b      # False  — different objects (two separate list allocations)
a is c      # True   — c is just another label for the same list

# Small integer caching — CPython caches ints -5 to 256
# WHY this range: CPython pre-allocates a pool of common small integers at startup to avoid
# repeated allocation; this is an implementation detail, not guaranteed by the language spec
x = 256; y = 256
x is y      # True   (cached — both names point to the same pre-allocated object)
x = 257; y = 257
x is y      # False  (not cached — two separate int objects were created, but x == y is still True)

# RULE: always use == for value comparison, use `is` only for None/True/False
# WHY `is None` not `== None`: None is a singleton — there is exactly one None object;
# `is` is faster (pointer comparison) and avoids accidental __eq__ overrides
if result is None:      # correct — checking for None
    ...
if result == None:      # works but style warning — use `is` for singletons
