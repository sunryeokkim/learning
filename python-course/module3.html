Module 03

Control Flow & Functions

Loops, comprehensions, functions, closures, recursion, and memoization — the control structures that form every algorithm.

Loops

for Loop — Every Useful Form

# ── Basic iteration ───
for x in [1, 2, 3]:
    print(x)

# ── Range ───
for i in range(5):          # 0, 1, 2, 3, 4
for i in range(2, 8):       # 2, 3, 4, 5, 6, 7
for i in range(10, 0, -2):  # 10, 8, 6, 4, 2   (countdown, step -2)
for i in range(n-1, -1, -1):# n-1 down to 0   (reverse index loop)
# WHY -1 as stop: range's stop is exclusive, so -1 is needed to include index 0

# ── enumerate — index + value together ───
fruits = ["apple", "banana", "cherry"]
for i, fruit in enumerate(fruits):
    print(i, fruit)           # 0 apple, 1 banana, 2 cherry

for i, fruit in enumerate(fruits, start=1):  # start index at 1
    print(i, fruit)           # 1 apple, 2 banana, 3 cherry
# WHY start=1: useful when displaying 1-based ranks or matching 1-indexed problem input

# ── zip — iterate multiple lists in parallel ───
names  = ["Alice", "Bob", "Charlie"]
scores = [95, 87, 92]
for name, score in zip(names, scores):
    print(f"{name}: {score}")

# zip stops at the shortest list — use zip_longest for full coverage:
from itertools import zip_longest
for a, b in zip_longest([1,2,3], [4,5], fillvalue=0):
    print(a, b)   # (1,4) (2,5) (3,0)
# WHY fillvalue=0: prevents None from propagating into arithmetic on the padded side

# ── Nested loops ───
for i in range(rows):
    for j in range(cols):
        process(grid[i][j])
# WHY outer=rows, inner=cols: row-major order; matches how Python stores 2-D lists in memory

# ── for...else — runs if loop finishes WITHOUT break ───
target = 7
for x in nums:
    if x == target:
        print("found!")
        break
else:
    # WHY else on for: the only way to reach here is if no break fired —
    # cleaner than a boolean flag variable to track "was target found?"
    print("not found")   # only runs if no break occurred

while Loop

# Standard while
i = 0
while i < n:
    process(arr[i])
    i += 1
# WHY while instead of for: use when the step size is variable or the
# termination condition depends on values computed inside the loop

# Two-pointer idiom
left, right = 0, len(arr) - 1
# Invariant: everything outside [left, right] has already been processed
while left < right:   # terminates because left increases OR right decreases each iteration
    # ... do something with arr[left], arr[right]
    left += 1
    right -= 1
# WHY left < right (not <=): when they meet there is no pair left to examine

# Read until condition
while (line := input()):   # walrus operator := assigns AND evaluates
    process(line)
# WHY walrus here: avoids calling input() twice (once to check, once to use);
# loop exits when input() returns an empty string, which is falsy

# while...else (same rule as for...else)
while stack:
    x = stack.pop()
    if x == target:
        break
else:
    # WHY else on while: reached only if the condition (stack non-empty) became
    # False naturally — i.e., stack drained without ever hitting the break
    print("never found target")

Comprehensions

Comprehensions are the most Python-idiomatic way to create collections. They are typically faster than equivalent for-loops because they are optimized at the bytecode level.

List Comprehension

Anatomy of a list comprehension
[ x**2 for x in range(10) if x % 2 == 0 ] ────── ─── ─ ── ────────── ── ─────────── expr var iterable filter (what (loop (source (only include to output) name) sequence) when True) Execution order: for each x in range(10) → test filter → evaluate expr x=0: 0%2==0 ✓ → 0**2 = 0 x=1: 1%2==0 ✗ → skipped x=2: 2%2==0 ✓ → 2**2 = 4 x=3: 3%2==0 ✗ → skipped x=4: 4%2==0 ✓ → 4**2 = 16 ... Result: [0, 4, 16, 36, 64]
The for and if clauses run left-to-right. The expression at the front is evaluated last, only when the filter passes.
# Syntax: [expression  for  variable  in  iterable  if  condition]

# Basic
squares = [x**2 for x in range(10)]         # [0, 1, 4, 9, 16, 25, 36, 49, 64, 81]

# With filter
evens = [x for x in range(20) if x % 2 == 0]  # [0, 2, 4, ..., 18]

# Transformation + filter
result = [x**2 for x in nums if x > 0]
# WHY filter before squaring: squaring negatives produces positives, so filter must
# happen at the source value, not the transformed value

# Flatten 2D list
matrix = [[1, 2, 3], [4, 5, 6], [7, 8, 9]]
flat = [x for row in matrix for x in row]   # [1, 2, 3, 4, 5, 6, 7, 8, 9]
# Note: outer loop first, inner loop second (reads like: "for each row, for each x in row")
# WHY this order: mirrors nested for-loop order; reversing the clauses would be a NameError

# Transpose matrix
transposed = [[row[i] for row in matrix] for i in range(len(matrix[0]))]
# WHY len(matrix[0]): number of columns drives the outer index; assumes a non-empty matrix
# Invariant: at each i, inner list collects the i-th element from every row → one new column

# Conditional expression inside comprehension
labels = ["pos" if x > 0 else "neg" for x in nums]
# WHY ternary before 'for' not after: this is an expression (always emits a value),
# not a filter — every element in nums gets a label, none are skipped

Dict and Set Comprehension

# Dict comprehension
freq = {c: "hello".count(c) for c in "hello"}  # {'h':1, 'e':1, 'l':2, 'o':1}
# WHY iterate over the string directly: each character becomes a key; duplicates
# are naturally de-duplicated because later assignments overwrite the same key

# Invert a dict (assumes values are unique)
d = {"a": 1, "b": 2, "c": 3}
inv = {v: k for k, v in d.items()}   # {1:'a', 2:'b', 3:'c'}
# WHY "assumes values are unique": if two keys share a value, the last one wins
# and the earlier mapping is silently lost — no error is raised

# Set comprehension
unique_lengths = {len(word) for word in ["cat", "dog", "elephant", "fox"]}
# {3, 8} — deduplicated
# WHY curly braces without colon: {expr} is a set, {k: v} is a dict; an empty
# set must be written set(), not {} (which produces an empty dict)

# Generator expression (lazy — does not build the full list)
# Use inside functions that consume iterables one at a time
total = sum(x**2 for x in range(1000))      # no intermediate list
# WHY no extra parens: when a generator is the sole argument, one set of parens suffices
any_neg = any(x < 0 for x in nums)          # short-circuits on first True
# WHY short-circuit matters: stops scanning as soon as answer is known — O(1) best case
all_pos = all(x > 0 for x in nums)          # short-circuits on first False
max_sq  = max(x**2 for x in nums)
# WHY prefer generator over list here: max only needs one pass; building a list
# first wastes O(n) memory that is immediately discarded

Functions

Parameters and Return Values

# Basic function
def add(a, b):
    return a + b

# Default arguments (evaluated ONCE at definition time)
def greet(name, prefix="Hello"):
    return f"{prefix}, {name}!"

greet("Alice")            # "Hello, Alice!"
greet("Bob", "Hi")        # "Hi, Bob!"
greet("Eve", prefix="Hey")# "Hey, Eve!" — keyword argument
# WHY keyword argument: makes call sites self-documenting and allows skipping
# earlier defaults when a later one needs to be overridden

# !! Mutable default argument TRAP
def append_to(x, result=[]):   # BAD — [] is created once and reused
    result.append(x)
    return result
# WHY reused: Python evaluates default expressions when the def statement runs,
# not each time the function is called — the same list object lives in the function's __defaults__

append_to(1)   # [1]
append_to(2)   # [1, 2]  ← NOT [2]! Same list is reused!

# Fix: use None as sentinel
def append_to(x, result=None):
    if result is None:   # WHY 'is None' not '== None': identity check is safer;
        result = []      # a custom object could override __eq__ to equal None
    result.append(x)
    return result

# Multiple return values (returns a tuple)
def min_max(arr):
    return min(arr), max(arr)   # comma creates a tuple — no parentheses required

lo, hi = min_max([3, 1, 4, 1, 5])  # lo=1, hi=5

# *args — variable positional arguments (collected as tuple)
def total(*nums):
    return sum(nums)
# WHY tuple not list: tuples are immutable; callers cannot accidentally mutate
# the args collection inside the function
total(1, 2, 3, 4)   # 10

# **kwargs — variable keyword arguments (collected as dict)
def show(**info):
    for k, v in info.items():
        print(f"{k} = {v}")
show(name="Alice", age=30)

# Unpacking into function calls
nums = [1, 2, 3]
print(*nums)         # print(1, 2, 3) — unpacks list as positional args
d = {"end": "\n"}
print("hi", **d)     # print("hi", end="\n") — unpacks dict as keyword args
# WHY ** unpack works here: print accepts **kwargs; the dict keys must match
# valid parameter names or a TypeError is raised

Lambda Functions

# lambda: anonymous function, single expression
square = lambda x: x**2
square(5)   # 25
# WHY lambda instead of def: useful when the function is short-lived and only
# passed as an argument — giving it a name via def adds noise without benefit

# Primary use: sort keys
nums = [3, -1, 4, -1, 5]
nums.sort(key=lambda x: abs(x))        # sort by absolute value
# WHY key= instead of comparing elements directly: Python calls key() once per
# element and compares the results — avoids repeated computation during comparisons
nums.sort(key=lambda x: (x%2, x))      # odds first, then evens; each group sorted
# WHY x%2 first in tuple: Python compares tuples element-by-element, left to right;
# x%2 == 1 for odd, 0 for even — so odds sort before evens

pairs = [(1,3), (2,1), (3,2)]
pairs.sort(key=lambda p: p[1])         # sort by second element
pairs.sort(key=lambda p: (-p[1], p[0]))# desc by second, then asc by first
# WHY negate for descending: Python's sort is always ascending; negating a numeric
# key reverses the effective order without a separate reverse=True that would flip both

# Sorting with multiple tie-breaking criteria (very common in intervals)
intervals = [(1,4), (1,2), (2,3), (1,3)]
intervals.sort(key=lambda x: (x[0], x[1]))  # start asc, then end asc
# WHY explicit tuple key: Python's sort is stable, but the secondary criterion
# only applies when x[0] ties — the tuple makes that intent explicit and readable

# Using lambda with filter/map (usually prefer comprehensions)
evens = list(filter(lambda x: x%2==0, nums))   # prefer: [x for x in nums if x%2==0]
doubled = list(map(lambda x: x*2, nums))        # prefer: [x*2 for x in nums]
# WHY prefer comprehensions: filter/map return lazy iterators (must wrap in list()),
# and the lambda adds syntactic overhead; comprehensions are more readable in Python

Closures and Nested Functions

A closure is a function that captures variables from its enclosing scope. In coding problems, nested functions are useful for DFS/backtracking where you want to share state without global variables.

# Nested function captures outer variables (closure)
def find_paths(grid):
    rows, cols = len(grid), len(grid[0])
    result = []           # captured by inner function; shared across all dfs frames

    def dfs(r, c, path):
        if r == rows-1 and c == cols-1:
            result.append(path[:])   # WHY path[:]: path is mutated in-place during
            return                   # backtracking; slice copies the current snapshot
        for dr, dc in [(0,1),(1,0)]:   # WHY only right and down: grid paths go toward
            nr, nc = r+dr, c+dc        # bottom-right; no revisiting means no up/left
            if 0 <= nr < rows and 0 <= nc < cols:   # WHY both bounds: guards against
                path.append(grid[nr][nc])            # negative indices wrapping around
                dfs(nr, nc, path)
                path.pop()   # WHY pop after recurse: restores path to its state before
                             # this branch — backtracking invariant

    dfs(0, 0, [grid[0][0]])   # WHY seed with grid[0][0]: starting cell is part of path
    return result

# nonlocal — modify a variable in enclosing (non-global) scope
def count_nodes(root):
    count = [0]             # trick: use list to avoid nonlocal
    def dfs(node):
        if not node: return
        count[0] += 1       # WHY index [0]: mutating the list contents is allowed
        dfs(node.left)      # without nonlocal; reassigning 'count' itself would not be
        dfs(node.right)
    dfs(root)
    return count[0]

# Cleaner with nonlocal (Python 3)
def count_nodes(root):
    count = 0
    def dfs(node):
        nonlocal count       # tells Python we want the outer 'count'
        # WHY nonlocal needed: without it, 'count += 1' creates a new local variable
        # in dfs's scope (because += is an assignment), shadowing the outer count
        if not node: return  # base case: None node — nothing to count; without this
                             # dfs(node.left) would crash when node is a leaf
        count += 1
        dfs(node.left)
        dfs(node.right)
    dfs(root)
    return count

Recursion

A recursive function calls itself with a simpler subproblem until it reaches a base case. Every recursive call consumes stack space — Python's default limit is ~1000 frames.

Call stack for factorial(4)
factorial(4) └─ return 4 * factorial(3) └─ return 3 * factorial(2) └─ return 2 * factorial(1) └─ return 1 * factorial(0) └─ BASE CASE: return 1 Unwinds: 1 → 1*1=1 → 2*1=2 → 3*2=6 → 4*6=24
Call stack frames for factorial(4) — grows down, unwinds up
▼ Growing (each call pushes a frame) ┌──────────────────────────────────────────┐ ← top of call stack │ factorial(4) waiting for factorial(3)… │ ├──────────────────────────────────────────┤ │ factorial(3) waiting for factorial(2)… │ ├──────────────────────────────────────────┤ │ factorial(2) waiting for factorial(1)… │ ├──────────────────────────────────────────┤ │ factorial(1) waiting for factorial(0)… │ ├──────────────────────────────────────────┤ │ factorial(0) BASE CASE → returns 1 │ ← deepest frame └──────────────────────────────────────────┘ ▲ Unwinding (each return pops a frame and resumes the one above) factorial(0) returns 1 factorial(1) returns 1 × 1 = 1 factorial(2) returns 2 × 1 = 2 factorial(3) returns 3 × 2 = 6 factorial(4) returns 4 × 6 = 24 ← final answer Each frame holds: local variable n, and the return address. Python default stack depth ≈ 1000 frames → set higher with sys.setrecursionlimit().
The call stack is LIFO — last in, first out. Deep recursion (n = 10,000) overflows it; use iteration or sys.setrecursionlimit() for DP problems.
# Three requirements for correct recursion:
# 1. Base case (stops the recursion)
# 2. Recursive case (moves toward base case)
# 3. Trust the recursion (assume recursive call returns correct result)

def factorial(n):
    if n <= 1:       # 1. base case
        return 1
    # WHY n <= 1 (not n == 0): covers both 0! = 1 and 1! = 1, and also guards
    # against accidental negative input that would recurse infinitely
    return n * factorial(n - 1)  # 2. recursive case, 3. trust it returns (n-1)!
    # WHY n-1 converges: each call reduces n by exactly 1 — guaranteed to reach base case

# Fibonacci — naive (exponential time, don't use without memoization)
def fib(n):
    if n <= 1: return n   # WHY return n (not 1): fib(0)=0 and fib(1)=1 are both
                          # equal to n itself; a single expression covers both
    return fib(n-1) + fib(n-2)   # ← recomputes fib(n-2) multiple times!
    # WHY exponential: fib(n-2) is recomputed inside fib(n-1)'s subtree, and again
    # here — the call tree has ~2^n nodes without caching

# Increase recursion limit for deep problems
import sys
sys.setrecursionlimit(200000)   # needed for n up to ~100,000
# WHY 200000 not unlimited: Python grows the C stack for each frame; too high
# a limit can cause a hard OS-level segfault instead of a clean RecursionError

# Convert recursion to iteration to avoid stack overflow
def factorial_iterative(n):
    result = 1
    for i in range(2, n+1):   # WHY start at 2: multiplying by 1 is a no-op;
        result *= i            # starting at 2 skips it without changing correctness
    return result
    # Invariant: after each iteration, result == i! (i factorial)

Memoization — Caching Recursive Results

Memoization stores the result of a function call so that if the same arguments are seen again, we return the cached result immediately instead of recomputing. This transforms overlapping-subproblem recursion from exponential to polynomial time.

Without vs. with memoization for fib(5)
WITHOUT memo: WITH memo: fib(5) fib(5) ├─ fib(4) ├─ fib(4) │ ├─ fib(3) │ ├─ fib(3) │ │ ├─ fib(2) │ │ ├─ fib(2) │ │ │ ├─ fib(1) ✓ │ │ │ ├─ fib(1) ✓ │ │ │ └─ fib(0) ✓ │ │ │ └─ fib(0) ✓ │ │ └─ fib(1) ✓ │ │ └─ fib(1) ✓ (cached) │ └─ fib(3) ← recomputes! │ └─ fib(2) ✓ (cached) │ ├─ fib(2) ← recomputes! └─ fib(3) ✓ (cached) │ ... Total calls: 9 Total calls: 15 (grows as 2^n) Time: O(n), Space: O(n) 
from functools import lru_cache, cache

# @cache (Python 3.9+) — unbounded memoization, simplest to use
@cache
def fib(n):
    if n <= 1: return n   # base case: fib(0)=0, fib(1)=1; without this the
                          # recursion never terminates (no smaller subproblem exists)
    return fib(n-1) + fib(n-2)
    # WHY @cache makes this O(n): each unique n is computed exactly once; the
    # decorator intercepts the call and returns the stored result on repeat calls

fib(100)   # fast — each value computed once and cached

# @lru_cache(maxsize=None) — same as @cache, available from Python 3.2
@lru_cache(maxsize=None)
def fib(n):
    if n <= 1: return n
    return fib(n-1) + fib(n-2)

# @lru_cache(maxsize=128) — bounded cache (LRU eviction)
# WHY bounded: caps memory use; LRU (Least Recently Used) evicts the entry that
# was accessed longest ago, preserving recently needed results

# Multiple parameters
@cache
def dp(i, j, state):
    if i == 0: return 0   # WHY base case at i==0: represents the empty prefix;
                          # without it the recursion on i-1 would go negative forever
    # ... transition
    return min(dp(i-1, j, state), dp(i, j-1, state))
    # WHY min: choosing the lower-cost sub-result — both i and j decrease, so
    # every branch converges toward the i==0 base case

# Cache with mutable arguments — use tuples, not lists
# lru_cache requires HASHABLE arguments
@cache
def dp(i, mask):           # mask is an int — hashable ✓
    # WHY bitmask int: represents a subset of n items in O(1) space; each bit
    # position corresponds to one item — far more cache-friendly than a frozenset
    ...

# If you need to pass a list, convert to tuple first
def solve(nums):
    @cache
    def dp(i, remaining):   # remaining is a tuple — hashable ✓
        # WHY nest dp inside solve: gives dp a fresh cache per call to solve,
        # preventing stale results if solve is called multiple times with different nums
        ...
    return dp(0, tuple(nums))
    # WHY tuple(nums): lists are unhashable so @cache would raise a TypeError;
    # tuple conversion happens once here, not on every recursive call

# Clear cache between test cases (if reusing the function)
fib.cache_clear()   # WHY needed: @cache persists for the lifetime of the function object;
                    # without clearing, a second test case inherits stale cached values
print(fib.cache_info())   # CacheInfo(hits=..., misses=..., maxsize=None, currsize=...)
💡
Top-down vs Bottom-up DP

Memoization (top-down) lets you write the natural recursive solution and just add @cache. Bottom-up DP (filling a table) avoids recursion overhead and stack overflow. For most problems, start with @cache to verify correctness, then convert to bottom-up if needed for performance.
