Module 04

Searching & Sorting

Binary search in all its forms — including searching on the answer — and Python's sorting tools from built-ins to custom comparators.

Binary Search

Binary search finds a target in a sorted array in O(log n) time by repeatedly halving the search space. The key insight: after every comparison, you can eliminate half the remaining candidates.

Recognition signals: The problem says "sorted array", "find in O(log n)", or the input is clearly ordered (rotated, range of integers, etc.). Also watch for "minimum/maximum value satisfying a condition" — that is binary search on the answer in disguise.

Key insight / invariant: At every point in the algorithm, if target exists in the array it must be in arr[lo..hi]. Each iteration provably shrinks that window by at least half. When lo > hi the window is empty, proving the target is absent.

Why sorted is required: The decision "go left or go right" depends on comparing arr[mid] to target. That comparison is only meaningful when the array is ordered — in an unsorted array, a larger value at mid tells you nothing about which side holds the target.

Pitfall: Binary search is only correct on truly sorted (or monotone) data. Applying it to an unsorted array silently produces wrong answers with no error.

Binary search trace — finding 7 in [1,3,5,7,9,11,13]
Array: [1, 3, 5, 7, 9, 11, 13] 0 1 2 3 4 5 6 Step 1: lo=0, hi=6, mid=3, arr[3]=7 == target → FOUND at index 3 Another example — finding 9: Step 1: lo=0, hi=6, mid=3, arr[3]=7 < 9 → search right half Step 2: lo=4, hi=6, mid=5, arr[5]=11 > 9 → search left half Step 3: lo=4, hi=4, mid=4, arr[4]=9 == 9 → FOUND at index 4

The Template — Exact Search

Why <= in the while condition: Both bounds are inclusive, so a single-element window (lo == hi) is still a valid, unexamined candidate. Using < would skip that last element and miss targets at the final position.

Why mid + 1 and mid - 1 (not mid): We already know arr[mid] != target at the point we update lo or hi, so mid itself can be safely excluded. Including mid in the next window would not cause a wrong answer here, but it could create an infinite loop in variants where mid is kept (see the "Find First True" template).

What happens on exit: The loop exits when lo > hi, which means the search window is empty — every element has been eliminated. Returning -1 at that point is correct by the invariant.

def binary_search(arr, target):
    lo, hi = 0, len(arr) - 1   # both bounds inclusive; search space is arr[lo..hi]
    # Invariant: if target exists in arr, it is within arr[lo..hi]

    while lo <= hi:             # <= because a window of size 1 (lo==hi) is still valid
        mid = (lo + hi) // 2   # integer midpoint; Python ints never overflow
        if arr[mid] == target:
            return mid          # found — exact match; invariant satisfied trivially
        elif arr[mid] < target:
            lo = mid + 1        # arr[mid] too small; exclude mid and everything left of it
        else:
            hi = mid - 1        # arr[mid] too large; exclude mid and everything right of it
    # Loop exits when lo > hi — search space is empty; target is not present
    return -1
ℹ️
Why mid = (lo + hi) // 2?

In C++/Java this can overflow for large values. In Python integers are arbitrary precision so there's no overflow, but it's still the standard form. Some prefer lo + (hi - lo) // 2 out of habit from other languages.

Leftmost / Rightmost Insertion — bisect_left and bisect_right

These are the most important binary search variants. They find the leftmost or rightmost position where you could insert target while keeping the array sorted.

Mental model for bisect_left: Imagine sliding a card labelled target from the left into the sorted deck. Stop at the first position where inserting it keeps the deck sorted — that is the leftmost position where arr[i] >= target. All indices to the left have strictly smaller values.

Mental model for bisect_right: Same card, but push it as far right as possible before the order breaks — stop at the first position where arr[i] > target. All indices to the left have values <= target.

Counting occurrences: Because bisect_left marks the start and bisect_right marks one-past-the-end of the run of equal values, their difference is exactly the count of occurrences. This is O(log n) versus O(n) for a linear scan.

Pitfall: After bisect_left, always guard with i < len(arr) and arr[i] == target before concluding the value is present — the function returns a valid insertion index even when the target is absent.

import bisect

arr = [1, 3, 3, 3, 5, 7]
#      0  1  2  3  4  5

# bisect_left — leftmost position where target could be inserted
# Result: index of first element >= target (or len(arr) if all elements < target)
bisect.bisect_left(arr, 3)    # → 1  (insert before first 3; arr[1]==3 is the match)
bisect.bisect_left(arr, 4)    # → 4  (4 absent; arr[4]==5 is the first element > 4)
bisect.bisect_left(arr, 0)    # → 0  (0 smaller than everything; insert at front)
bisect.bisect_left(arr, 9)    # → 6  (9 larger than everything; insert at end)

# bisect_right — rightmost position (also called bisect())
# Result: index of first element strictly > target (one past the last occurrence)
bisect.bisect_right(arr, 3)   # → 4  (insert after last 3; arr[4]==5 is first > 3)
bisect.bisect_right(arr, 4)   # → 4  (same as left when value absent — no equal elements to skip)

# Common use: check if value exists in sorted array
def exists(arr, target):
    i = bisect.bisect_left(arr, target)
    # bisect_left finds where target WOULD be; check that slot actually holds target
    return i < len(arr) and arr[i] == target

# Count occurrences of value in sorted array — O(log n)
def count_occurrences(arr, target):
    left  = bisect.bisect_left(arr, target)   # index of first occurrence (or insert point)
    right = bisect.bisect_right(arr, target)  # index one past last occurrence
    return right - left   # all indices in [left, right) contain target; slice length = count

# Floor — largest element <= target
def floor(arr, target):
    # bisect_right gives one-past the last element <= target, so step back one
    i = bisect.bisect_right(arr, target) - 1
    return arr[i] if i >= 0 else None   # i == -1 means every element is > target

# Ceiling — smallest element >= target
def ceiling(arr, target):
    # bisect_left gives the first position where arr[i] >= target
    i = bisect.bisect_left(arr, target)
    return arr[i] if i < len(arr) else None   # i == len means every element is < target

# Insert while maintaining sorted order
bisect.insort(arr, 4)   # inserts 4 in the correct position — O(log n) to find, O(n) to shift

Manual Variants — Find First True

Recognition signals: "Find the leftmost index where...", "find the first element satisfying...", or any problem where you need a boundary between two states in a sorted/ordered structure.

The monotone requirement: This template only works when the condition is monotone — the array of boolean values looks like [False, False, ..., True, True, ...] with no True-to-False transition. If the condition is not monotone, binary search will converge to the wrong boundary.

Why hi = len(arr) (not len(arr) - 1): If every element is False, the correct return value is len(arr) (meaning "not found" / "insert at end"). Using len(arr) - 1 as the initial upper bound would prevent lo from ever reaching len(arr), returning a wrong index.

Why hi = mid (not mid - 1): When condition(arr[mid]) is True, mid itself is a candidate answer. Setting hi = mid - 1 would lose it. In contrast, we set lo = mid + 1 in the False branch because mid is definitely not the answer.

Why lo < hi (not lo <= hi): The loop maintains the invariant that the answer is in [lo, hi]. When lo == hi, we have pinpointed a single candidate — no more iterations needed. Using <= would produce one extra iteration that could cause an infinite loop when hi = mid and lo == hi == mid.

Pitfall: When searching for "last True" and biasing mid upward ((lo + hi + 1) // 2), forgetting the +1 causes an infinite loop when lo + 1 == hi — mid rounds down to lo, lo = mid = lo never advances.

The condition must be monotone: all False on the left, all True on the right. Binary search finds the exact boundary in O(log n).

The most powerful binary search template: find the boundary where a condition switches from False to True. This solves "find leftmost X", "find first element ≥ k", and "search on the answer" problems.

# Pattern: array looks like [False, False, ..., False, True, True, ..., True]
# Find the FIRST index where condition is True
# Requires: condition is monotone (no True-to-False transition anywhere)

def first_true(arr, condition):
    lo, hi = 0, len(arr)   # hi = len(arr), not len(arr)-1, so lo can reach len(arr) when all False
    # Invariant: answer (first True index) is in the half-open range [lo, hi]
    while lo < hi:           # stop when window collapses to a single point (lo == hi)
        mid = (lo + hi) // 2
        if condition(arr[mid]):
            hi = mid         # mid is True and could be the FIRST True; keep it in range
        else:
            lo = mid + 1     # mid is False; it cannot be the answer, exclude it
    # lo == hi; this is the first True index, or len(arr) if every element was False
    return lo

# Example: first index where arr[i] >= target (equivalent to bisect_left)
arr = [1, 3, 5, 7, 9]
lo, hi = 0, len(arr)        # hi = len(arr) so we can return "not found" = len(arr)
while lo < hi:
    mid = (lo + hi) // 2
    if arr[mid] >= 6:
        hi = mid             # arr[mid] satisfies condition; might be leftmost, so keep mid
    else:
        lo = mid + 1         # arr[mid] < 6; too small, discard left half including mid
# lo = 3 (arr[3]=7 is first element >= 6)

# Pattern: Find LAST index where condition is True (search from right)
def last_true(arr, condition):
    lo, hi = -1, len(arr) - 1   # lo starts at -1 so it can signal "all False"
    # Invariant: answer (last True index) is in [lo, hi]; lo tracks the best True seen so far
    while lo < hi:
        mid = (lo + hi + 1) // 2   # +1 biases mid UPWARD to avoid lo==mid infinite loop
        if condition(arr[mid]):
            lo = mid               # mid is True and could be the LAST True; push lo up
        else:
            hi = mid - 1           # mid is False; last True must be left of mid
    # lo == hi; this is the last True index, or -1 if no True was found
    return lo

Binary Search on the Answer — The Most Powerful Pattern

Many optimization problems can be solved by binary searching on the answer value rather than an array index. The key insight: if you can check whether a given answer is feasible in O(f(n)), you can find the optimal answer in O(f(n) log(answer_range)).

Recognition signals: "Find the minimum/maximum value such that...", "what is the smallest capacity/speed/length that allows...", or "minimize the maximum / maximize the minimum". Whenever the feasibility of a candidate answer is easy to check in a linear pass, binary search on the answer applies.

Key insight — why feasibility must be monotone: If some answer value X is feasible, then any value larger than X must also be feasible (for a minimization problem). This monotone structure — False for small values, True from some threshold onward — is exactly what binary search needs. Verify this property before applying the pattern.

How to set lo and hi: lo is the smallest answer that could possibly be correct (a natural lower bound, e.g. 1, or max(arr)). hi is the largest it could ever need to be (a natural upper bound, e.g. sum(arr)). Being conservative is fine — extra iterations cost only O(log factor).

Pitfall: Choosing bounds that exclude the real answer (e.g., setting lo = 0 when 0 is never valid but the real minimum is 1) or bounds that are too loose (e.g., hi = 10**18 when the answer fits in 10**6) both work correctly but the latter wastes log-factor iterations. More dangerously, if the feasibility check is not truly monotone, the algorithm will still converge — just to the wrong value, silently.

# Template for "find minimum X such that condition(X) is True"
# Requires: is_feasible is monotone — once True for some value, True for all larger values

lo, hi = minimum_possible_answer, maximum_possible_answer
# Invariant: answer is in [lo, hi]; lo is always infeasible, hi is always feasible (or boundary)
while lo < hi:
    mid = (lo + hi) // 2
    if is_feasible(mid):   # can we achieve answer = mid?
        hi = mid           # mid works; it could be optimal, so keep it in range
    else:
        lo = mid + 1       # mid doesn't work; optimal must be strictly larger
# lo == hi is the minimum feasible answer

# ─────────────────────────────────────────────────────────────────
# EXAMPLE: "Koko Eating Bananas"
# piles = [3, 6, 7, 11], h = 8 hours
# Find minimum eating speed k such that Koko can eat all bananas in h hours
import math

def min_eating_speed(piles, h):
    def can_finish(speed):
        # ceil(p/speed) hours needed per pile; sum must fit in h total hours
        hours = sum(math.ceil(p / speed) for p in piles)
        return hours <= h
    # lo=1: can't eat slower than 1 banana/hr (speed=0 is undefined)
    # hi=max(piles): eating at this speed finishes any pile in exactly 1 hour — always feasible
    lo, hi = 1, max(piles)
    while lo < hi:
        mid = (lo + hi) // 2
        if can_finish(mid):
            hi = mid        # speed mid is sufficient; try slower
        else:
            lo = mid + 1    # speed mid is too slow; must go faster
    # lo is the minimum speed that finishes all piles within h hours
    return lo

# ─────────────────────────────────────────────────────────────────
# EXAMPLE: "Capacity To Ship Packages Within D Days"
# Find minimum ship capacity to deliver all packages in D days

def ship_within_days(weights, days):
    def can_ship(capacity):
        current_load, day_count = 0, 1
        for w in weights:
            if current_load + w > capacity:
                day_count += 1   # start a new day
                current_load = 0
            current_load += w
        return day_count <= days

    lo = max(weights)   # lower bound: capacity must fit the heaviest single package
    hi = sum(weights)   # upper bound: enough capacity to ship everything on day 1 (always feasible)
    # Monotone: if capacity C ships in <=days days, capacity C+1 also can (more room per day)
    while lo < hi:
        mid = (lo + hi) // 2
        if can_ship(mid):
            hi = mid        # capacity mid works; try smaller
        else:
            lo = mid + 1    # capacity mid too small; need more
    # lo is the minimum feasible capacity
    return lo

Sorting

Python's Sort — Timsort

Python uses Timsort — a hybrid of merge sort and insertion sort. It's O(n log n) worst case and O(n) on already-sorted data. It is stable: equal elements keep their original relative order.

Recognition signals: Any problem that says "sort then sweep" or requires a canonical ordering before applying a greedy or two-pointer technique. Also applies when the problem asks you to rank, group, or process elements in a specific order.

Why stability matters: A stable sort preserves the relative order of elements that compare as equal. This lets you sort by a secondary key first, then sort by the primary key — the two-pass approach works because equal primary keys keep their secondary order intact. In Python you rarely need this trick since key= tuples handle multi-criteria sorting in one pass, but stability is still important when sorting objects that only expose a partial comparison.

The key= mental model: Python extracts the key value for each element once, sorts the keys (which must be comparable), and reorders elements to match. This is the Schwartzian transform — O(n) key computations, O(n log n) comparisons on cheap keys. Think of it as "decorate → sort → undecorate" done automatically.

Pitfall: Do not pass a comparison function directly — Python 3 removed cmp=. Use key= for almost everything. Only reach for functools.cmp_to_key when the comparison cannot be expressed as a key (e.g., non-transitive orderings or string concatenation comparisons like largest-number problems).

# sort() — in-place, returns None
nums = [3, 1, 4, 1, 5, 9]
nums.sort()                    # [1, 1, 3, 4, 5, 9]
nums.sort(reverse=True)        # [9, 5, 4, 3, 1, 1]

# sorted() — returns new list, original unchanged; works on any iterable
nums = [3, 1, 4, 1, 5, 9]
asc = sorted(nums)             # [1, 1, 3, 4, 5, 9]  — nums unchanged
desc = sorted(nums, reverse=True)

# Sort any iterable (not just lists)
sorted("banana")               # ['a', 'a', 'a', 'b', 'n', 'n']
sorted({3: 'c', 1: 'a', 2: 'b'})  # [1, 2, 3] — sorts dict keys (values ignored)

# ── The key parameter ─── (callable: element → comparable sort key)
# Python calls key(element) once per element, sorts by the returned keys
words = ["banana", "fig", "cherry", "apple"]
words.sort(key=len)                      # sort by word length
words.sort(key=lambda w: w[-1])          # sort by last character
words.sort(key=lambda w: (len(w), w))    # primary: length, secondary: alphabetical

# Sort by multiple criteria — tuple comparison is lexicographic by default
students = [("Alice", 90), ("Bob", 85), ("Eve", 90)]
students.sort(key=lambda s: (-s[1], s[0]))  # negate score for descending; name ascending
# [("Alice", 90), ("Eve", 90), ("Bob", 85)]

# Stability example — equal primary keys retain their original relative order
nums = [(1, "b"), (2, "a"), (1, "a")]
nums.sort(key=lambda x: x[0])
# [(1, "b"), (1, "a"), (2, "a")]  — both 1s were already in this order; stability preserved it

Sorting Patterns for Coding Problems

Recognition signals: "Merge overlapping intervals", "arrange to maximize/minimize some property", "sort by custom rule", or "elements appear in a fixed priority order". Sorting is usually the first step in a greedy algorithm.

Key insight — custom key functions avoid cmp_to_key: Most custom orderings can be expressed as a tuple key. For example, sorting strings so that AB comes before BA requires comparing a+b vs b+a — that cannot be expressed as a simple key, so it is one of the few genuine uses of functools.cmp_to_key. For everything else, a tuple key is simpler, faster, and more readable.

Pitfall — modifying a list while sorting: Never sort a list of mutable objects (like [start, end] lists) by a key that changes during iteration. In the merge-intervals pattern, mutation happens after sorting is complete, so it is safe. Also note that intervals.sort() on lists-of-lists sorts lexicographically, which is usually correct for [start, end] pairs.

# ── Interval scheduling (sort by start, then end) ───
# Tuple/list comparison is lexicographic, so (start, end) sorts by start first
intervals.sort()   # tuples sort lexicographically — (start, end)

# ── Merge intervals ───
intervals.sort(key=lambda x: x[0])   # sort by start time; ties broken by end time implicitly
merged = [intervals[0]]
for start, end in intervals[1:]:
    if start <= merged[-1][1]:         # current interval overlaps the last merged one
        merged[-1][1] = max(merged[-1][1], end)   # extend the merged interval's end
    else:
        merged.append([start, end])    # no overlap; start a new merged interval

# ── Relative sort / custom order ───
order = ["grape", "apple", "banana"]
order_map = {v: i for i, v in enumerate(order)}   # maps each item to its desired rank
items = ["banana", "apple", "grape", "kiwi"]
# Items in order_map get their rank; unknown items get rank len(order) (sorted last)
items.sort(key=lambda x: order_map.get(x, len(order)))

# ── Sort by frequency, then value ───
nums = [1, 1, 2, 2, 2, 3]
freq = Counter(nums)                             # build frequency map in O(n)
nums.sort(key=lambda x: (-freq[x], x))          # negate freq for descending; value for ascending tie-break
# [2, 2, 2, 1, 1, 3]

# ── Counting sort — O(n+k) when range is bounded ───
# Use when values are non-negative integers in a small known range [0, k]
# Faster than Timsort when k << n*log(n); not applicable for floats or strings
def counting_sort(arr, max_val=None):
    if max_val is None:
        max_val = max(arr)          # O(n) scan to find range; skip if you already know it
    count = [0] * (max_val + 1)    # bucket for each value 0..max_val
    for x in arr:
        count[x] += 1              # tally occurrences; O(n)
    # Reconstruct: for each value x, emit it count[x] times — O(n+k) total
    return [x for x, c in enumerate(count) for _ in range(c)]
