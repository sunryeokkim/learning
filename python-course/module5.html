Algorithm Patterns

The four patterns that unlock the majority of array and string problems: two pointers, sliding window, prefix sums, and backtracking.

Two Pointers

Use two index variables (pointers) that move through the data, usually toward each other or in the same direction. Reduces O(n²) brute force to O(n) by avoiding redundant work.

Recognition signals: The problem involves a sorted array and asks you to find a pair or triplet satisfying a sum condition. Or: modify an array in-place, partitioning elements by some property (e.g. remove duplicates, move zeroes). Or: detect a cycle in a linked list. Any time you see "find two elements that…" on sorted data, two pointers is the first pattern to reach for.

Key insight — why it beats O(n²) brute force: A naive nested loop tests every (i, j) pair. Two pointers exploit the sorted order to eliminate an entire row or column of the search space at each step. If the current sum is too large, every pair involving the right element is also too large (since the array is sorted), so we can discard all of them in one move. This gives O(n) eliminations total, compared to O(n²) individual checks.

Pitfall: The opposite-ends pattern only works correctly on sorted input. On unsorted data, you cannot guarantee that moving a pointer in one direction monotonically increases or decreases the sum, so the invariant breaks.

Pattern 1: Opposite Ends (Sorted Array)

Start one pointer at the left end (smallest value) and one at the right end (largest value). Move them inward based on whether the current result is too small or too large.

Why sorting first works — the exchange argument: After sorting, nums[l] is the minimum available candidate and nums[r] is the maximum. If their sum is too large, the only way to decrease it is to move r left — moving l right would only increase the sum further. This one-directional reasoning is the exchange argument: at every decision point, exactly one move is safe, so we never backtrack and never revisit a position.

Mental model: Think of l and r as the "floor" and "ceiling" of the current search range. The invariant is: if any valid pair exists, it must lie within nums[l..r]. Each step provably eliminates at least one endpoint while preserving that invariant, so the loop terminates in at most n steps.

# Problem: Two Sum II (sorted array)
# Given sorted nums, find two indices whose values sum to target.
def two_sum_sorted(nums, target):
    l, r = 0, len(nums) - 1   # l points to the smallest candidate, r to the largest
    # Invariant: if a valid pair exists, it lies within nums[l..r].
    # Proof: we only advance l when nums[l] is provably too small for every r' <= r,
    # and only retreat r when nums[r] is provably too large for every l' >= l.
    while l < r:               # l == r would mean using the same element twice
        s = nums[l] + nums[r]
        if s == target:
            return [l + 1, r + 1]   # 1-indexed as required by the problem
        elif s < target:
            # Sum is too small. nums[r] is already the largest possible right partner
            # for this l, so no right partner can fix it — advance l instead.
            l += 1
        else:
            # Sum is too large. nums[l] is already the smallest possible left partner
            # for this r, so no left partner can fix it — retreat r instead.
            r -= 1
    # Termination: l and r converge; every pair has been considered exactly once.

# Problem: Container With Most Water
# height[i] is the wall height at position i; find two walls enclosing the most water.
def max_area(height):
    l, r = 0, len(height) - 1   # start with the widest possible container
    best = 0
    while l < r:
        # Area = width * height of shorter wall.
        # Width = r - l; shorter wall is the bottleneck because water spills over it.
        area = (r - l) * min(height[l], height[r])
        best = max(best, area)
        if height[l] < height[r]:
            # Moving r left would only shrink the width while the bottleneck stays
            # height[l]. Moving l right is the only action that could raise the floor.
            l += 1
        else:
            # Symmetric: height[r] is the bottleneck; moving l right keeps a short wall.
            # Only retreating r can possibly find a taller left bound.
            r -= 1
    return best

# Problem: 3Sum — find all unique triplets summing to 0
def three_sum(nums):
    nums.sort()   # sort so the two-pointer inner loop works and duplicates are adjacent
    result = []
    for i in range(len(nums) - 2):   # i is the "anchor"; leave room for l and r
        # Skip duplicate anchors: if nums[i] == nums[i-1] we already explored this anchor
        # and would generate the same triplets again.
        if i > 0 and nums[i] == nums[i-1]:
            continue
        l, r = i + 1, len(nums) - 1   # standard two-pointer on the remaining subarray
        # Invariant: if a valid pair (l, r) exists for anchor nums[i], it is in nums[l..r].
        while l < r:
            s = nums[i] + nums[l] + nums[r]
            if s == 0:
                result.append([nums[i], nums[l], nums[r]])
                # Skip duplicate l values: advancing l over equal values avoids recording
                # the same triplet with a different l index.
                while l < r and nums[l] == nums[l+1]: l += 1   # skip dup lefts
                while l < r and nums[r] == nums[r-1]: r -= 1   # skip dup rights
                l += 1; r -= 1   # move both pointers inward to search for the next pair
            elif s < 0:
                l += 1   # triplet sum too small; increase by picking a larger left value
            else:
                r -= 1   # triplet sum too large; decrease by picking a smaller right value
    return result

Pattern 2: Same Direction (Fast and Slow)

Both pointers move in the same direction but at different speeds. Used for in-place array modification and cycle detection.

Mental model — slow = write head, fast = read head: Think of slow as a tape-recorder head that only advances when it has something worth writing, and fast as the playback head that scans every element. The gap between them is the "garbage" region that has already been processed and will be overwritten. Because fast always leads slow, the write never clobbers an unread value.

Floyd's cycle insight: If a linked list has a cycle, a pointer moving one step at a time will loop forever while one moving two steps at a time will lap it — they are guaranteed to meet inside the cycle. The exact meeting point has a beautiful property: the distance from the head to the cycle start equals the distance from the meeting point to the cycle start (modulo the cycle length), which is why resetting one pointer to head and walking both at speed 1 finds the cycle entry exactly.

Pitfall: Always check fast and fast.next before advancing fast by two steps — skipping either null-guard causes a runtime error on lists with an even number of nodes.

# Problem: Remove Duplicates from Sorted Array
# slow = write head (next position to write a unique value into)
# fast = read head (scans every element looking for new unique values)
# Because the array is sorted, duplicates are always adjacent, so a new unique
# value is detected simply by comparing fast to slow (the last written value).
def remove_duplicates(nums):
    if not nums: return 0
    slow = 0   # index of the last unique value written; also the write head
    for fast in range(1, len(nums)):
        # fast always leads slow, so writing to nums[slow+1] never clobbers an unread value
        if nums[fast] != nums[slow]:   # found a new unique value not yet written
            slow += 1                  # advance write head to the next open slot
            nums[slow] = nums[fast]    # commit the new unique value in-place
    # Invariant at termination: nums[0..slow] contains all unique values in order.
    return slow + 1   # new logical length (indices 0..slow inclusive)

# Problem: Move Zeroes
# Move all zeroes to the end while maintaining the relative order of non-zero elements.
# slow = next position to place a non-zero element (write head)
# fast = current element being examined (read head)
def move_zeroes(nums):
    slow = 0   # all positions before slow contain the non-zero elements already placed
    for fast in range(len(nums)):
        if nums[fast] != 0:
            # Copy non-zero value to the write head; safe because fast >= slow always
            nums[slow] = nums[fast]
            slow += 1   # advance write head; it now points at the next open slot
    # At this point nums[0..slow-1] = all non-zero values in original order.
    # Fill the remaining tail with zeroes.
    while slow < len(nums):
        nums[slow] = 0
        slow += 1

# Problem: Floyd's Cycle Detection (Linked List)
# slow moves 1 step per iteration, fast moves 2 steps.
# Key insight: if a cycle exists, fast will lap slow inside the cycle.
# The relative speed between them is 1 step/iter inside the cycle,
# so they must meet in at most cycle_length iterations after fast enters the cycle.
def has_cycle(head):
    slow = fast = head
    # Guard: fast.next must exist because fast is about to take two steps
    while fast and fast.next:
        slow = slow.next        # slow advances 1 step
        fast = fast.next.next   # fast advances 2 steps
        if slow == fast:
            return True   # they met inside the cycle
    # Termination without meeting: fast reached the end, so the list is acyclic.
    return False

# Find start of cycle — extends Floyd's to locate the exact entry node.
def detect_cycle(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow == fast:
            # Mathematical result: distance(head → cycle_start) ==
            # distance(meeting_point → cycle_start) mod cycle_length.
            # So resetting slow to head and walking both at speed 1 makes them
            # arrive at the cycle start simultaneously.
            slow = head
            while slow != fast:
                slow = slow.next   # both move exactly 1 step
                fast = fast.next
            return slow   # both pointers are now at the cycle start node
    return None   # no cycle found

Sliding Window

Maintain a "window" (contiguous subarray/substring) and slide it through the data. The window expands when we need more elements and shrinks when a constraint is violated. This gives O(n) instead of O(n²).

Recognition signals: The problem asks about a contiguous subarray or substring (longest, shortest, maximum sum, at most k distinct…). If you see "contiguous" and a constraint that must hold across the subarray, sliding window is almost certainly the right tool. Fixed-size: the window length k is given. Variable-size: you're optimizing the window length subject to some validity condition.

The amortized O(n) argument: The naive approach re-examines all elements in the window every time it slides — O(nk) or O(n²). Sliding window avoids this because each element enters the window exactly once (when right reaches it) and leaves exactly once (when left passes it). The total work across all expand and shrink steps is therefore O(n), regardless of how many times the window changes size.

Fixed vs variable: Use a fixed-size window when k is given explicitly. Use a variable-size window when you are optimizing the window length — the right pointer drives expansion while the left pointer is advanced reactively only when the window becomes invalid.

Fixed-Size Window

The window size k is known in advance. Initialize the sum over the first k elements, then slide one position at a time using the add-right / subtract-left update formula.

The add-right subtract-left update: When the window slides one position to the right, only two elements change: nums[right] enters the window and nums[right - k] leaves it. The update is therefore window_sum += nums[right] - nums[right - k] — a single O(1) operation instead of re-summing k elements. Over n - k slides, this gives O(n) total versus O(nk) for recomputation.

Pitfall: Off-by-one on the outgoing index. The element leaving the window when right moves to index i is at index i - k (not i - k + 1). Drawing one concrete example before coding always catches this.

# Maximum sum subarray of size k
# Strategy: seed the running sum with the first window, then slide one step at a time.
def max_sum_subarray(nums, k):
    window_sum = sum(nums[:k])   # O(k) to build the initial window [0..k-1]
    best = window_sum            # the first window is a candidate answer
    for i in range(k, len(nums)):
        # The add-right / subtract-left update: the window shifts one position right.
        # nums[i] enters on the right; nums[i - k] exits on the left.
        # This O(1) update replaces an O(k) re-sum — the key efficiency gain.
        window_sum += nums[i]       # add the new right element (index i)
        window_sum -= nums[i - k]   # remove the outgoing left element (index i-k)
        best = max(best, window_sum)
    # Termination: all n-k+1 windows of size k have been evaluated.
    return best

# Average of subarrays of size k — same sliding pattern, divided by k each step.
def find_averages(nums, k):
    result = []
    window_sum = sum(nums[:k])    # seed with the first k elements
    result.append(window_sum / k) # record average for window [0..k-1]
    for i in range(k, len(nums)):
        # Combined update: add incoming, subtract outgoing, in one expression.
        window_sum += nums[i] - nums[i - k]   # O(1) update — no recomputation
        result.append(window_sum / k)         # each division is O(1)
    # Invariant: result[j] = average of nums[j..j+k-1] for all valid j.
    return result

Variable-Size Window

The window grows when the constraint is satisfied and shrinks when it's violated. The right pointer always moves forward; the left pointer moves forward only when needed.

The expand/shrink contract: Right moves forward unconditionally on every iteration — it is the engine of progress. Left only moves forward to repair a violated constraint. This asymmetry is the key: right drives, left corrects. The window at any moment is the largest valid window ending at right.

Why left never goes backward: Once an element has been expelled from the left of the window, re-including it can only make the constraint harder to satisfy (e.g., it would re-introduce a duplicate or add to a count that was already too high). So any valid window with left' < left has already been considered — we never need to revisit it. This is the two-pointer amortization argument: left is monotonically non-decreasing, so its total movement across the entire loop is at most n steps, giving O(n) overall.

Pitfall: Forgetting to shrink the window before recording the best answer. If you record outside the while loop but the window is still invalid, you'll capture a bad state. The pattern is: expand (add right), shrink until valid (advance left), then record.

# General variable-size window template
# right drives expansion (moves every iteration); left drives shrinking (moves reactively)
left = 0
window_state = ...    # could be Counter, running sum, set, etc. — tracks the window's "state"

for right in range(len(s)):
    # EXPAND: unconditionally include s[right] in the window
    window_state = update_add(window_state, s[right])

    # SHRINK: repair the constraint by evicting from the left until the window is valid.
    # Using 'while' (not 'if') handles cases where multiple evictions are needed.
    while window_is_invalid(window_state):
        window_state = update_remove(window_state, s[left])
        left += 1   # left never goes backward — amortized O(n) total across all iterations

    # RECORD: window [left, right] is now the largest valid window ending at right.
    best = max(best, right - left + 1)

# ─────────────────────────────────────────────────────────────────
# Problem: Longest Substring Without Repeating Characters
def length_of_longest_substring(s):
    seen = {}       # char → index of its most recent occurrence in s
    l = best = 0   # l = left boundary of the current valid window
    for r, c in enumerate(s):
        # Check whether c is inside the current window (not just in seen historically)
        if c in seen and seen[c] >= l:
            # c is a duplicate within [l..r-1]; jump l past the previous occurrence.
            # This is O(1) shrinking: we know exactly how far left must move.
            l = seen[c] + 1
        seen[c] = r   # always update to the latest index; old index is now stale
        best = max(best, r - l + 1)   # window size = r - l + 1
    # Invariant at each step: s[l..r] contains no repeated characters.
    return best

# Problem: Longest Substring with At Most K Distinct Characters
# Window invariant: count contains only characters currently in s[l..r],
# and len(count) <= k must hold after each shrink phase.
def longest_k_distinct(s, k):
    from collections import defaultdict
    count = defaultdict(int)   # count[c] = frequency of c in current window
    l = best = 0
    for r, c in enumerate(s):
        count[c] += 1   # expand: add the new right character to the frequency map
        # Shrink until the window has at most k distinct characters.
        # Each iteration removes the leftmost char; if its count hits 0, it's no longer
        # in the window so we delete it from count (maintaining the len(count) invariant).
        while len(count) > k:
            count[s[l]] -= 1
            if count[s[l]] == 0:
                del count[s[l]]   # evict completely so len(count) reflects distinct count
            l += 1
        best = max(best, r - l + 1)
    return best

# Problem: Minimum Window Substring
# Find the smallest contiguous window in s that contains every character of t
# (with the required multiplicities).
def min_window(s, t):
    from collections import Counter
    need = Counter(t)          # need[c] = how many of c we still need in the window
    have, total = 0, len(need) # total = number of distinct chars in t we must satisfy
    # 'have' counts how many distinct chars are currently satisfied (count >= need).
    l = 0
    best = (float('inf'), 0, 0)   # (window_length, left_index, right_index)
    window = {}   # window[c] = frequency of c in current window s[l..r]

    for r, c in enumerate(s):
        window[c] = window.get(c, 0) + 1
        # Check if adding c has just satisfied one more required character.
        # We only increment 'have' at the exact threshold — not for every occurrence.
        if c in need and window[c] == need[c]:
            have += 1   # this char is now fully satisfied

        # Shrink from the left while the entire window is valid (have == total).
        # We shrink because we want the MINIMUM window — remove unnecessary left chars.
        while have == total:
            if r - l + 1 < best[0]:
                best = (r - l + 1, l, r)   # found a smaller valid window — record it
            # Evict s[l]: update window freq and check if we've un-satisfied a requirement.
            window[s[l]] -= 1
            if s[l] in need and window[s[l]] < need[s[l]]:
                have -= 1   # dropping below threshold — this char is no longer satisfied
            l += 1   # slide left boundary right; loop will check validity again
    # Termination: every starting position has been tried as the left boundary.
    return s[best[1]:best[2]+1] if best[0] != float('inf') else ""

Prefix Sums

Precompute cumulative sums so that any range sum query is answered in O(1) instead of O(n).

The offset-by-one convention — why prefix[0] = 0: The prefix array is intentionally one element longer than the input, with prefix[0] = 0 as a sentinel. This makes the range-sum formula uniform: sum(l..r) = prefix[r+1] - prefix[l] works for every l, including l = 0, without a special case. Without the sentinel, you'd need an if-statement for l = 0 — the sentinel eliminates that branch entirely.

Range sum derivation: prefix[i] holds the sum of nums[0..i-1]. So prefix[r+1] = nums[0..r] and prefix[l] = nums[0..l-1]. Their difference is exactly nums[l..r]. This is the fundamental identity that makes all prefix-sum tricks work.

When to use prefix sums vs sliding window: Use a sliding window when the subarray must satisfy a contiguous-validity constraint that you can check incrementally (e.g., at most k distinct characters). Use prefix sums when you need random-access range queries — especially when queries are non-contiguous, given offline, or you need to count subarrays satisfying an exact sum (the hashmap trick with prefix_count[prefix - k]). Sliding window cannot easily answer "how many subarrays sum exactly to k" because you can't binary-search on a running sum.

Recognition signals: "Range sum query", "subarray sum equals k", "count subarrays with property", "2D rectangle sum". Any problem where you need to answer multiple sum queries on the same array, or count subarrays by their total, is a prefix-sum candidate.

Prefix sum construction and range query
nums: [ 3, 1, 4, 1, 5, 9, 2, 6] idx: 0 1 2 3 4 5 6 7 prefix: [ 0, 3, 4, 8, 9, 14, 23, 25, 31] idx: 0 1 2 3 4 5 6 7 8 prefix[i] = nums[0] + nums[1] + ... + nums[i-1] Range sum of nums[2..5] (inclusive): = prefix[6] - prefix[2] = 23 - 4 = 19 ✓ (4 + 1 + 5 + 9 = 19) 
# Build prefix sum — O(n)
# prefix[i] = nums[0] + nums[1] + ... + nums[i-1]
# prefix[0] = 0 is the sentinel: it represents the empty prefix (no elements).
# This sentinel is why the range-sum formula works uniformly for l = 0.
def build_prefix(nums):
    n = len(nums)
    prefix = [0] * (n + 1)   # one extra slot for the sentinel at index 0
    for i in range(n):
        # Build incrementally: each prefix is the previous prefix plus nums[i].
        # Note the offset: prefix[i+1] corresponds to the cumulative sum up to nums[i].
        prefix[i + 1] = prefix[i] + nums[i]
    return prefix

# Range sum query [l, r] inclusive — O(1)
# Derivation: prefix[r+1] = sum(nums[0..r])
#             prefix[l]   = sum(nums[0..l-1])
# Difference  = sum(nums[l..r])  — the l-1 elements cancel out.
def range_sum(prefix, l, r):
    return prefix[r + 1] - prefix[l]

# Python one-liner using itertools.accumulate — identical semantics to build_prefix above.
from itertools import accumulate
prefix = [0] + list(accumulate(nums))   # prepend the sentinel 0, then cumulative sums

# ─────────────────────────────────────────────────────────────────
# Problem: Subarray Sum Equals K
# Count how many contiguous subarrays have sum exactly equal to k.
# Key insight: subarray nums[j..i] has sum k iff prefix[i+1] - prefix[j] = k,
# i.e., prefix[j] = prefix[i+1] - k. We look this up in a hash map in O(1).
def subarray_sum(nums, k):
    from collections import defaultdict
    # prefix_count[s] = number of times we have seen prefix sum s so far
    prefix_count = defaultdict(int)
    # Initialize with prefix_count[0] = 1: the empty prefix (before any element)
    # has sum 0. This handles subarrays that start at index 0 correctly.
    prefix_count[0] = 1

    prefix = result = 0
    for x in nums:
        prefix += x   # running prefix sum after including x
        # How many earlier prefix values equal (prefix - k)?
        # Each such occurrence corresponds to a subarray ending here with sum k.
        result += prefix_count[prefix - k]
        # Record the current prefix sum AFTER querying — order matters!
        # Recording before querying would accidentally count zero-length subarrays.
        prefix_count[prefix] += 1

    return result

# Problem: Contiguous Array (equal number of 0s and 1s)
# Reframe: replace 0 with -1. Equal 0s and 1s ↔ subarray sum = 0.
# Then the problem becomes: find the longest subarray with sum 0,
# which is equivalent to finding the farthest apart pair of equal prefix sums.
def find_max_length(nums):
    # seen[s] = the earliest index i at which the running prefix sum equalled s.
    # Sentinel: prefix sum 0 first occurs at index -1 (before the array starts).
    seen = {0: -1}
    prefix = best = 0
    for i, x in enumerate(nums):
        prefix += 1 if x == 1 else -1   # +1 for 1, -1 for 0
        if prefix in seen:
            # We've seen this prefix sum before at index seen[prefix].
            # The subarray from seen[prefix]+1 to i has sum 0 → equal 0s and 1s.
            best = max(best, i - seen[prefix])
        else:
            # Record only the FIRST occurrence — we want the longest span,
            # so an earlier left boundary is always at least as good as a later one.
            seen[prefix] = i
    return best

# 2D Prefix Sum
# prefix[i][j] = sum of all grid cells in the rectangle grid[0..i-1][0..j-1].
# Indices are offset by 1 (same sentinel trick as 1D) so the boundary conditions
# are uniform: prefix[0][*] = prefix[*][0] = 0.
def build_2d_prefix(grid):
    m, n = len(grid), len(grid[0])
    # Allocate (m+1) x (n+1) so that row 0 and col 0 serve as zero-sentinels.
    prefix = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            # Inclusion-exclusion: start with the full rectangle up to (i,j),
            # then subtract the top strip and left strip (each counted once),
            # then add back the top-left corner (subtracted twice, so add once).
            prefix[i][j] = (grid[i-1][j-1]     # current cell (0-indexed in grid)
                           + prefix[i-1][j]     # sum of the rectangle above
                           + prefix[i][j-1]     # sum of the rectangle to the left
                           - prefix[i-1][j-1])  # re-add corner double-subtracted above
    return prefix

# Rectangle sum from (r1,c1) to (r2,c2) inclusive — O(1)
# Same inclusion-exclusion in reverse: take the large rectangle,
# subtract the top strip, subtract the left strip, add back the doubly-subtracted corner.
def rect_sum(prefix, r1, c1, r2, c2):
    return (prefix[r2+1][c2+1]   # full rectangle ending at bottom-right
           - prefix[r1][c2+1]    # subtract the top strip (rows above r1)
           - prefix[r2+1][c1]    # subtract the left strip (cols left of c1)
           + prefix[r1][c1])     # add back the top-left corner (subtracted twice)

Backtracking

Backtracking builds solutions incrementally and abandons (backtracks) a partial solution as soon as it determines the path cannot lead to a valid complete solution. It's a systematic DFS through the solution space.

DFS tree metaphor: Every backtracking problem implicitly defines a tree. The root is the empty partial solution. Each node's children are the choices available at that step. Leaves are either complete solutions (record them) or dead ends (prune and return). Backtracking is simply a DFS over this tree — you go deep, hit a dead end or a solution, and then climb back up to try the next sibling branch.

Pruning — cutting branches early: The power of backtracking over brute-force enumeration is that you can detect impossible paths before fully exploring them. If the remaining candidates are all larger than the remaining target, you can break immediately rather than trying each one. Every pruned branch can save an exponential subtree of work. Sorting the candidates first is the most common enabler of early pruning.

The always-copy-state rule: When you record a complete solution, always copy the current path with path[:] (or list(path)). If you append the path object directly, every recorded "solution" will end up pointing at the same mutable list, which will be empty by the time the function returns — a subtle bug that produces a list of empty lists instead of the actual solutions.

i vs i+1 — reuse vs no-reuse: When you recurse with start = i + 1, each element can be chosen at most once (subsets, combinations). When you recurse with start = i, the same element can be chosen again (combination sum with repetition). This single argument controls all reuse semantics.

Recognition signals: "Find all…", "generate all…", "enumerate all valid…", "count the number of ways…". Whenever the problem asks you to explore all configurations of a discrete set — subsets, permutations, combinations, valid placements — backtracking is the template.

Backtracking tree for subsets of [1, 2, 3]
[] / | \ [1] [2] [3] / \ \ [1,2] [1,3] [2,3] / [1,2,3] All nodes (including intermediate) = subsets: [], [1], [1,2], [1,2,3], [1,3], [2], [2,3], [3] → 2^3 = 8 subsets 
# Universal backtracking template
# The DFS tree: root = empty state, each edge = one choice, leaves = complete solutions.
def backtrack(state, start, choices):
    # 1. Base case — record if the current state is a complete solution.
    if is_complete(state):
        results.append(state[:])   # ALWAYS copy: state is mutable and will be undone
        return

    # 2. Try each available choice starting from 'start'.
    # 'start' prevents re-choosing elements already committed earlier in this path.
    for i in range(start, len(choices)):
        choice = choices[i]

        # 3. Prune: skip this branch early if the choice cannot lead to a valid solution.
        # Pruning here avoids descending into a subtree we'd just backtrack from anyway.
        if not is_valid(state, choice):
            continue

        # 4. Make the choice — extend the current partial state.
        state.append(choice)

        # 5. Recurse deeper. i+1 = each element used at most once; i = reuse allowed.
        backtrack(state, i + 1, choices)

        # 6. Undo the choice — restore state to what it was before step 4.
        # This is the "backtrack" step: the for-loop's next iteration will try
        # the next sibling choice from a clean state.
        state.pop()

# SUBSETS — collect all 2^n subsets of nums (no duplicates in nums)
# Every node in the DFS tree is a valid subset — we record at every call, not just leaves.
def subsets(nums):
    result = []
    def bt(start, path):
        # Record the current partial path as a valid subset (including the empty set).
        result.append(path[:])   # copy because path will be mutated in subsequent calls
        for i in range(start, len(nums)):
            path.append(nums[i])
            # i+1: the next call can only choose elements after index i,
            # so each element appears at most once per subset and order is preserved.
            bt(i + 1, path)
            path.pop()   # undo to restore path for the next sibling branch
    bt(0, [])
    return result

# SUBSETS with duplicates — sort so equal values are adjacent, then skip duplicate branches.
# Without skipping, nums=[1,2,2] would produce [1,2] twice (once per '2').
def subsets_with_dups(nums):
    nums.sort()   # makes duplicate values adjacent so the skip condition is simple
    result = []
    def bt(start, path):
        result.append(path[:])
        for i in range(start, len(nums)):
            # Skip a duplicate value if we've already branched on the same value
            # at this same DFS level (i > start, not i > 0, to allow the same value
            # at different depths of the tree).
            if i > start and nums[i] == nums[i-1]:
                continue   # prune: this branch would duplicate an already-explored subtree
            path.append(nums[i])
            bt(i + 1, path)
            path.pop()
    bt(0, [])
    return result

# PERMUTATIONS — all n! orderings of nums.
# Unlike subsets, every element must appear; 'used' tracks which are already in the path.
def permutations(nums):
    result = []
    def bt(path, used):
        if len(path) == len(nums):
            result.append(path[:])   # leaf: path is a complete permutation
            return
        for i in range(len(nums)):
            if used[i]: continue   # skip elements already committed to this path
            used[i] = True         # mark as in-use for deeper recursive calls
            path.append(nums[i])
            bt(path, used)
            path.pop()
            used[i] = False        # unmark: restore for sibling branches
    bt([], [False] * len(nums))
    return result

# COMBINATIONS — choose exactly k elements from [1..n] without repetition.
# Pruning: if fewer than (k - len(path)) elements remain, no complete combo is possible.
def combinations(n, k):
    result = []
    def bt(start, path):
        if len(path) == k:
            result.append(path[:])   # leaf: we've chosen exactly k elements
            return
        # Upper bound pruning: last valid starting index is n - remaining_needed + 1.
        # If i > n - remaining_needed + 1, there aren't enough elements left to reach k.
        remaining_needed = k - len(path)
        for i in range(start, n - remaining_needed + 2):   # +2 because range is exclusive
            path.append(i)
            bt(i + 1, path)   # i+1: no reuse, each number chosen at most once
            path.pop()
    bt(1, [])
    return result

# COMBINATION SUM — choose numbers from candidates (with reuse allowed) summing to target.
# Sort candidates so we can break early when a candidate exceeds the remaining sum.
def combination_sum(candidates, target):
    candidates.sort()   # enables the break-early pruning inside the loop
    result = []
    def bt(start, path, remaining):
        # Base case: remaining hits exactly 0 — current path is a valid combination.
        if remaining == 0:
            result.append(path[:])
            return
        for i in range(start, len(candidates)):
            # Pruning: because candidates is sorted, all subsequent candidates are also
            # larger than remaining. No need to check them — break the entire loop.
            if candidates[i] > remaining:
                break
            path.append(candidates[i])
            # i (not i+1): the same candidate can be chosen again in the next level,
            # which is what "with reuse" means. This is the only difference from combinations.
            bt(i, path, remaining - candidates[i])
            path.pop()
    bt(0, [], target)
    return result
💡
Backtracking Optimization Tips
  • Sort first — enables early termination and duplicate skipping
  • Prune early — add if remaining < 0: return or if candidates[i] > remaining: break
  • Skip duplicates — if i > start and nums[i] == nums[i-1]: continue
  • Always copy state — result.append(path[:]) not result.append(path)
